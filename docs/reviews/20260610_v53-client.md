## Summary

The "thin client" contract is **genuinely honored where it matters**: no HP, mana, inventory, gold, loot, or damage outcome is ever computed client-side — all flow exclusively from server `state_delta` ops (verified across the 14 op-types in `_apply_delta`). Panel wiring is clean: UI panels emit typed signals and `main.gd` connects them, with no panel reaching back via `get_parent()`. That is the right architecture.

But `main.gd` is a **4,410-line god object** doing everything from login to camera to 12-panel construction to ~25 `bot_*` CI methods, and the four big panels (inventory/shop/stash) carry **verbatim duplicated** item-loading and icon-rendering code. The riskiest client logic — delta application and prediction/reconciliation — has **zero unit tests**.

**Client sub-scores:** Architecture-boundaries 8.5 · Architecture-cohesion 5.0 · Technical 7.5 · Maintainability 5.5 · Documentation 6.0.

## 1. Architecture

- **`main.gd` is a god object. [High]** 4,410 lines spanning: network lifecycle (`_ready`/`_process`/connect, ~223–758), protocol dispatch (`_handle_message`/`_apply_snapshot`/`_apply_delta`, 761–1077), entity/loot/projectile/wall node factories (1079–1230, 3279–3800), camera + prediction (1291–1303, 1622–1628), all input routing (1519–2100), construction of **all 12 UI panels in one 60-line `_build_scene`** (2388–2460), menu/lobby/settings state machine (341–656), bot/CI scaffolding (~25 `bot_*` methods, 3837–4360), visual-replay playback (~200 lines), and autoplay (~80 lines).
- **Panel wiring is clean. [Strength]** Panels expose `signal intent_requested(intent_type, payload)` and `main` connects at construction (2422–2440); no panel takes a back-reference to main. This is the one part of the client that scales well.
- **Thin-client outcome contract holds. [Strength]** `_apply_delta` (882–945) and `_upsert_entity` (1086–1103) mutate state only from server ops; prediction (1104–1107) moves `predicted_pos` then reconciles immediately to `server_pos`. No combat/loot authority client-side.
- **But rules *data* is loaded 5× independently. [Med]** `items.v0.json` / `item_templates.v0.json` / `item_presentations.v0.json` are read separately in `main.gd:3586`, `inventory_panel.gd:1162`, `shop_panel.gd:1076`, `stash_panel.gd:595`, `consumable_bar.gd`. A rules-key rename silently breaks whichever loader you forgot.
- **Procedural-only UI.** Only 3 `.tscn` files; all 12 panels are built in code. Avoids scene bloat but makes visual editing impossible and concentrates construction in `main.gd`.

## 2. Technical

- **Unknown server messages are silently dropped. [High]** `_handle_message` (`main.gd:763`) has no `_:` default arm. A new server message type produces zero output — protocol evolution becomes invisible to debugging. One-line fix: `_: push_warning(...)`.
- **Unguarded delta payload access. [Med]** `_apply_delta` reads `c["entity"]` (`:888`) and `c["item_instance_id"]` (`:900`) directly; a malformed/partial op null-derefs and crashes. Use `c.get(..., {})`.
- **Hardcoded contract IDs. [Med]** `MAGIC_BOLT_ID="magic_bolt"` (`:45`), `ARCHER_MONSTER_DEF_ID="dungeon_archer"` (`:75`), `INTERACTABLE_ACTIVATION_RANGE=1.5` (`:44`) bake server-side identities/physics constants with no cross-check; silent misfire if the server renames.
- **Stale default literals. [Low]** `PLAYER_START_HP=10` (`:43`) and `player_mana=10` (`:660`, bare literal) are used as teardown resets; a delayed first snapshot could let input gates read a wrong value.
- **Typing is inconsistent. [Low]** Class vars in `main.gd` are mostly typed; locals drift to inferred (`inventory_panel.gd:1169`), and `bot_controller.gd:23` uses fully-untyped `var _main = null` duck-typing.
- **Headless test fragility. [Med]** `bot_controller.gd:7-14` correctly documents that `Input.warp_mouse()` is a no-op headless and uses a direct-dispatch fallback — but `click_floor` paths still read `player_anchor.global_position`, so they only work with an active scene tree. Documented, but fragile.

## 3. Maintainability

- **New panel ⇒ ≥5 edits to `main.gd`. [High]** Each panel must be declared (~`:193`), built in `_build_scene` (`:2388`), signal-wired, fed in `_refresh_inventory_ui` (`:1272`), hidden in `_close_gameplay_panels` (`:1656`), and cleared in `_teardown_gameplay_state` (`:653`). No registry/factory/scene-composition.
- **Verbatim duplication across panels. [High]**
  - `_draw_item_icon` (~45-line shape-match block) copy-pasted in `inventory_panel.gd:491`, `shop_panel.gd:323`, `stash_panel.gd:334`.
  - The 3-function item-rules loader block duplicated in 5 scripts.
  - `show_status(text, error)` independently reimplemented in `shop_panel.gd:160` and `stash_panel.gd:141`.
- **Bot scaffolding entangled with the game loop. [High]** ~220 lines of `autoplay_*`/`visual_replay_*`/`bot_*` state in `main.gd`; `get_bot_state` (`:3839`) exposes a ~50-key internal dict; `_process` (`:739`) branches `visual_replay → autoplay → handle_input`. Can't isolate test scaffolding from production rendering.
- **Riskiest code is untested. [Med]** Strong: `test_client_bot.gd` (55 tests), `test_animation.gd` (10). But **zero** unit coverage of `_apply_delta`, `_apply_snapshot`, `_handle_intent_rejected`, or prediction/reconciliation. `test_shop_panel.gd`/`test_stash_panel.gd` are render scenes, not headless assertions.
- **Terminology drift. [Low]** `character_flow` constants mix `CHARACTER_FLOW_LEGACY_SOLO` with `CHARACTER_FLOW_CREATE_GAME` — accumulated naming without cleanup.

## 4. Documentation

- **Panel files have no headers. [Med]** `inventory_panel.gd`, `shop_panel.gd`, `stash_panel.gd`, `skills_panel.gd` open straight to `class_name`. `main.gd`, `net_client.gd`, `animation_controller.gd` do carry ADR-referenced headers.
- **No function docstrings anywhere. [Med]** No `##` Godot doc annotations; e.g. `set_inventory_state` (`inventory_panel.gd:198`, 7 params) is undocumented.
- **`animation_controller.gd` is well-documented. [Strength]** Class header states the terminal > one-shot > locomotion priority machine clearly.
- **Contract touch-points partly commented. [Med]** `_apply_snapshot`/`_apply_delta` have scattered inline notes; `net_client.gd:138` references spec §4.5 — but the 14 delta op payload shapes are undocumented.
- **Vestigial `reconciliation_delta`. [Low]** Computed at `main.gd:1104` but never read — dead debug metric.

## Top 5 client refactors

1. **Add a `default` arm + `push_warning` to `_handle_message`** (`main.gd:763`) and `.get()`-guard the delta payload reads (`:888,:900`). One-line safety wins for protocol evolution. *(trivial/high)*
2. **Extract an `ItemRulesLoader` autoload + `ItemIconRenderer` helper.** Kills the loader-×5 and icon-×3 duplication and 5+ future edit sites. *(low/high)*
3. **Isolate bot/CI scaffolding** (~600–700 lines: `autoplay_*`, `visual_replay_*`, `bot_*`) into a `BotMainProxy` child node, leaving the production `_process` path branch-free. *(med/high)*
4. **Introduce a `UIOrchestrator` child node** owning panel construction + show/hide/refresh/teardown, so a new panel is a registration, not 5 edits to `main.gd`. *(med/med)*
5. **Add headless unit tests for `_apply_delta`/`_apply_snapshot`/reconciliation** with fixture deltas — the highest-risk, currently-untested client code. *(med/med)*
