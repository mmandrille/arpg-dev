## 1. Executive Summary

**Overall score: 8.4 / 10.**

The repo is healthier than it was at v53. The most important prior review findings were acted on:
the sim has an input-handler registry, determinism lint runs in CI, item rules load through a shared
Godot singleton, unknown client messages now warn, delta application has unit coverage, health-regen
naming has explicit validation, and golden regeneration exists.

The system is ready for the smallest data-driven content-library follow-up: a **skills-only manifest
loader with no behavior change**. Skills now have a compact schema-backed mechanics catalog, a
separate presentation catalog, generic server validation, client singleton loading, validation
coverage, and bot/golden proof. The review does **not** recommend jumping straight to item/class
library splitting because those surfaces still span more loaders, refs, UI rendering, and economy
contracts.

The top current risks are concentrated and actionable: runtime sim construction still has a panic
path, realtime fanout reads sim state outside the loop lock, `main.gd` and `validate_shared.py`
remain large coordination files, and protocol/schema versions keep accumulating without retirement
rules.

## 3. Cross-Cutting Themes

### Theme 1 - The v53 review had real payoff

The repo did not just document risks; it closed several. `handlers.go` now owns intent registration,
CI runs `make lint-determinism`, `make regen-golden` exists, `ItemRulesLoader` centralizes item
catalog loading, `test_delta_apply.gd` covers state application, and `CLAUDE.md` now points to the
current v59 baseline and 15 Hz tick.

### Theme 2 - Skills are the correct manifest proving ground

The data-driven content note says file-path dictionaries must be loader indexes, not runtime model
IDs. A skills-only manifest can prove deterministic declared-order loading, duplicate-ID rejection,
relative path resolution, schema validation, presentation cross-checks, and stable sorted exposure
without touching combat behavior or persistence.

### Theme 3 - Broad item/class manifests should wait

Items are much wider than skills: fixed item definitions, equipment templates, visuals,
presentations, loot tables, shops, appraisal/comparison rows, inventory/stash panels, and icon
rendering. The first manifest should avoid that blast radius.

### Theme 4 - The next monolith risk moved into loaders and validators

`sim.go` is smaller, but `rules.go` and `validate_shared.py` are now where content growth will land.
If the manifest slice simply adds more code to those files, it repeats the old pattern. The slice
should introduce a small loader/validator seam as part of the proof.

## 4. Top 10 Recommendations

1. **Next slice:** implement `data-driven-content-library-manifest` for skills only, no behavior
   change, using the research note as the spec input.
2. Keep stable skill IDs (`magic_bolt`) as the gameplay model; file paths must not enter protocol,
   persistence, golden fixtures, or runtime state.
3. Put manifest validation in a focused helper/module rather than extending the long middle of
   `validate_shared.py`.
4. Add duplicate skill-ID and unknown manifest-group tests as first-class loader checks.
5. Replace runtime uses of `NewSim` panic construction with error-returning constructors.
6. Snapshot client levels under lock before realtime fanout.
7. Guard the remaining direct `equipped_update` slot access in client delta handling.
8. Extract item icon rendering before item manifest splitting.
9. Define a protocol-version retirement/archive policy before creating v9/v10.
10. Turn the manifest result into an ADR/as-built decision after the first slice lands.
