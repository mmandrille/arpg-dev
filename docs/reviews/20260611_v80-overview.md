## 3. Cross-cutting themes

### Theme 1 - Server facts, client readability

The current direction is correct: v80 uses `monster_aggro` as an authoritative server event and only adds a client-side `AGGRO` presentation at `client/scripts/main.gd:1110`.

### Theme 2 - Encounters are becoming structured

Pack counts, member radius, roles, assist radius, and elite leader markers now give future combat slices a better foundation than isolated random monster placement. Evidence: pack generation and composition live in `server/internal/game/dungeon_gen.go:609` and `server/internal/game/dungeon_gen.go:642`.

### Theme 3 - Large coordination files are still the tax

`client/scripts/main.gd` is 5297 lines, `tools/bot/run.py` is 4910 lines, `tools/validate_shared.py` is 3110 lines, and `server/internal/game/game_test.go` is 8247 lines. These files still work, but future slices should extract helpers when they touch these areas.

### Theme 4 - One v70 correctness finding remains worth doing

Realtime fanout still reads player current levels after unlocking the session loop. Evidence: lock release at `server/internal/realtime/session_loop.go:514` and sim read at `server/internal/realtime/session_loop.go:529`.

## 4. Top 10 recommendations

1. Snapshot realtime client levels under lock before `fanoutResult`.
2. Extract combat event presentation from `client/scripts/main.gd`.
3. Guard `env["payload"]` access in `client/scripts/main.gd`.
4. Derive client bot `ALL_STEP_TYPES` from wait/assert/action lists.
5. Split future dungeon-generation tests out of `game_test.go`.
6. Keep pack leader data internal until a UI/gameplay feature needs protocol exposure.
7. Split `tools/bot/run.py` into driver, assertions, and special co-op modules.
8. Split `tools/validate_shared.py` by validation domain.
9. Prefer semantic generated-content assertions over exact population counts.
10. Document the server prerequisite for direct `make client-smoke`, or steer full smoke through `make ci`.
