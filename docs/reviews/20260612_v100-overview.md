## 3. Cross-cutting themes

### Theme 1 - Damage typing should be a shared contract

The next slice should define canonical damage types, use `force` as fallback, and validate every non-force type in shared data. This keeps skills, weapons, monsters, bot proof, and future UI presentation aligned.

### Theme 2 - Resistances should be one system

Flying lightning resistance, quadruped lightning weakness, and undead poison immunity are all the same concept. Model resistance as a numeric multiplier/reduction in monster rules rather than creating special-case booleans.

### Theme 3 - Presentation can trail authority

The client does not need to own damage-type math. It can render existing damage events first, then later opt into type-specific colors/effects once the server payload is stable.

### Theme 4 - Keep the next combat work out of monoliths

File-size scan found `game_test.go` at 8766 lines, `sim.go` at 7566, `main.gd` at 6144, `tools/bot/run.py` at 5037, and `tools/validate_shared.py` at 3074. The v100 plan should put combat math and tests in focused files where practical.

## 4. Top 10 recommendations

1. Implement `force`, `cold`, `poison`, and `lightning` as canonical shared damage types.
2. Add `damage_type` to skills/weapons with `force` fallback for missing values.
3. Add monster `resistances` as data, with positive values reducing damage and negative values increasing damage.
4. Prove flyers resist lightning by 50% and quadrupeds take 50% more lightning damage in bot or focused Go tests.
5. Defer undead until the resistance contract is landed, then express poison immunity as `poison: 1.0`.
6. Add shared schema/validator checks for damage-type names and resistance ranges.
7. Keep combat formula tests in a focused file, not `game_test.go`.
8. Keep client changes presentation-only and tolerant of missing `damage_type`.
9. Add skeleton visuals through the existing asset manifest and monster visual catalog.
10. Run `make ci` after each committed slice and keep commits separate.

## 5. What "done well" looks like here

Damage type and resistance behavior is data-driven, deterministic, covered by shared validation and Go tests, and proven through a bot scenario. Undead then becomes a small content/presentation slice that reuses the same rules rather than adding bespoke poison-immunity logic.
