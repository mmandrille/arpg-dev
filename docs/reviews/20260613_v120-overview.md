# v120 Engineering Review Overview

Date: 2026-06-13
Baseline: post-v120 working tree, `make ci` green
Previous review: [`20260613_v110-overview.md`](20260613_v110-overview.md)

## Summary

The v111-v120 batch preserved the main architectural strengths: server authority for economy,
elite behavior, item upgrades, and unique item effects; data-driven rules for tunable gameplay; and
bot-backed proofs for player-facing flows. The repo remains in good shape for an actively expanding
game, but the same maintainability pressure identified in v100/v110 is still present in the largest
coordinator, simulator, validator, and bot files.

Overall score: 8.5/10.

## Scorecard

| Area | Score | Notes |
|------|-------|-------|
| Server authority | 9/10 | Market purchase/delivery, blacksmith upgrades, elite auras, and live unique effects stay server-owned. |
| Data ownership | 8.5/10 | v119/v120 improved unique/rule coverage; accidental tuning locks still need gradual cleanup. |
| Client architecture | 8/10 | Focused panels exist, but `main.gd` still coordinates too many town services and UI bridges. |
| Test strategy | 8.5/10 | Broad bot and CI coverage remain strong; largest test buckets should be split as domains mature. |
| Tooling/process | 8.5/10 | SDD and maintainability ratchets are active; the validator and bot runner need modular pressure relief. |

## What Improved

- Economy flows moved from backend-only contracts to usable Godot flows without giving clients
  mutation authority.
- The blacksmith slice closed the missing player-facing upgrade entrypoint by adding a town NPC and
  dedicated upgrade panel.
- Live unique drops now prove every enabled unique effect can roll through compatible equipment
  templates.
- v120 started paying down the balance-test brittleness by deriving a skills-panel test from shared
  skill rules instead of copying Magic Bolt tuning numbers.

## Cross-Cutting Risks

1. `client/scripts/main.gd` is still the main UI/service coordinator. Recent town services were
   implemented with focused panels, but the open/show/close bridge code remains centralized.
2. `server/internal/game/sim.go` and `server/internal/game/game_test.go` are still the broadest
   backend accumulation points. New gameplay domains should add focused files before extending them.
3. `tools/validate_shared.py` and `tools/bot/run.py` continue to carry many unrelated domains in
   one file. This is acceptable while the ratchet holds, but future catalog and scenario work should
   split by domain.
4. Tuning-friendly assertions are now proven in one Godot test. The rest of the repo still needs an
   audit for accidental copies of shared rule values.

## Recommended Next Inputs

- Extract a small town-service/panel bridge from `main.gd` before adding another NPC-driven panel.
- Split shared validation by catalog area, starting with unique items/effects or skills.
- Continue the tuning-friendly test pass against bot scenarios and Go tests that copy balance values.
- Prefer focused backend test files for new itemization/economy behavior instead of adding to
  `game_test.go`.
