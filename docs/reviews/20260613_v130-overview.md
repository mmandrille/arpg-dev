## 1. Executive summary

The v121-v130 batch moved the project forward without weakening its core boundary: the server still
owns item movement, market status, Ranger combat, and durable audit rows, while the client renders
and stages player intent. The strongest engineering improvement was follow-through on v120 review
findings: skill validation was split out, bot skill assertions now derive caps from rules, and town
service bridge logic moved out of `main.gd`.

The repo is ready for the requested unique-items batch after this review, with two cautions. First,
test-focused unique work should keep chest contents server-authored and deterministic. Second, the
next unique slices should avoid expanding the largest files by default: use focused validation,
bot, client, and store helpers where practical.

Review follow-up also refreshed the maintainability baseline for file-size drift already present at
the v130 boundary. Future slices should treat those refreshed counts as the new ratchet, not as
permission to keep growing the same files.

## 3. Cross-cutting themes

1. **Prior review items are being acted on.** v126 split skill validation, v127 extracted
   town-service bridge logic, and v125 continued the tuning-friendly test pass.
2. **Market ownership is now stronger but more complex.** Expiration, withdrawal, purchase,
   acceptance, refunds, and audit rows are in place; the next market work should extract repository
   boundaries before adding more transaction paths.
3. **Unique item work has a good runway.** Live unique effects are already server-authored, and the
   next player-facing test chest can be a deterministic lab surface rather than a new loot model.
4. **The largest files should not be the default landing zone.** `main.gd`, `game_test.go`,
   `tools/bot/run.py`, `tools/validate_shared.py`, and store market code are the places to avoid
   growing casually.

## 4. Top 10 recommendations

1. Build the purple town unique chest as a deterministic server-authored lab interactable.
2. Keep the chest contents generated from the enabled unique-effect catalog rather than duplicating
   effect IDs in tests.
3. Add a bot scenario that opens/picks from the chest and proves every enabled unique effect is
   obtainable.
4. Split unique item/effect validation out of `tools/validate_shared.py` before expanding unique
   catalog rules.
5. Add fixed named unique item packages after the test chest proves the player-facing loop.
6. Add a small unique inspection UI so players can see rarity and effect identity without raw JSON.
7. Extract market store methods from `repos.go` before the next advanced economy slice.
8. Clarify market expiration from summary/detail paths instead of relying only on active-list reads.
9. Keep new Ranger/unique backend tests in focused files instead of `game_test.go`.
10. Reword the stale `tuning-friendly-rule-tests` candidate in `PROGRESS.md` as continuing audit work.

## 5. What "done well" looks like here

The next unique-items slices should produce small end-to-end proofs: shared rule validation, server
authority, a deterministic world or bot lab, client rendering only where needed, and `make ci` green
before closeout. For the requested town chest specifically, done well means a visible purple chest
in town, contents derived from all currently enabled unique effects, protocol/bot proof that the
items are obtainable, and no client-side shortcut that invents items locally.
