`client/scripts/main.gd`, `client/scripts/bot_scenario_runner.gd`, and `tools/bot/run.py`.

## 3. Cross-cutting themes

1. **v130 findings were mostly closed.** Unique chest, named uniques, unique validation split,
   unique inspection UI, bot assertion split, CODEMAP, and market read freshness all shipped.
2. **The authority boundary is still clean.** The client renders and stages actions; the server owns
   market item movement, unique effects, stash contents, and replayable gameplay outcomes.
3. **The maintainability gate now has teeth.** `make ci` runs `make maintainability`, so new large
   files and stale baselines are blocked rather than advisory.
4. **The next risks are concentration risks.** Large files remain the default mental map unless
   future specs deliberately route work into focused modules.

## 4. Top 10 recommendations

1. Extract market listing/offer/expiration/audit methods from `server/internal/store/repos.go`.
2. Create a small client bot facade so `main.gd` stops growing test hook adapters.
3. Split `client/scripts/bot_scenario_runner.gd` by step/assertion domain.
4. Continue extracting runtime assertion domains from `tools/bot/run.py`.
5. Extract the next rules validator from `tools/validate_shared.py` when a new catalog grows.
6. Keep new combat/unique tests out of `game_test.go` unless the plan documents why.
7. Keep CODEMAP updated whenever a slice moves or creates domain files.
8. Preserve market audit writes in the same transaction as future market mutations.
9. Add visual scenario commands to specs/plans for any visible client presentation slice.
10. Use this review as `/next` input before starting the next feature batch.

## 5. What "done well" looks like here

The next slices should continue the current pattern: small specs, focused plans, server-owned
outcomes, data-driven rules, targeted tests first, `make ci` before commit, and CODEMAP/ratchet
updates when files move or grow. For maintenance slices, done well means shrinking or isolating one
large surface without changing player behavior.
