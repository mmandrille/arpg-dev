## 1. Executive summary

The v141-v149 batch did exactly what the v140 review asked for. Market persistence moved from the general store coordinator into focused market files. Session load and player lifecycle moved out of `sim.go`. `main.gd` shed bot adapter bodies into `BotFacade`. `BotScenarioRunner` now delegates catalog, wait, assertion, and action responsibilities. The result is a real trend bend, not just documentation. The maintainability ratchet now reports 33 grandfathered files / 65592 lines, down from 68778 at v140. The largest files are still large, but the next work is better targeted: `game_test.go`, the remaining `sim.go` domains, `main.gd` presentation ownership, Python runtime assertion dispatch, and `validate_shared.py`.

## 3. Cross-cutting themes

1. **The v140 recommendations were acted on.** Market store extraction, client bot facade, client bot runner split, Python assertion split, and further Python runtime splits all shipped between v141 and v149.
2. **The ratchet is changing behavior.** Shrink slices lowered baselines immediately, and `make maintainability` now records the downward trend.
3. **Compatibility wrappers were a useful migration pattern.** `main.gd` and `tools/bot/run.py` kept public call sites stable while moving bodies to focused modules.
4. **The next risk is test and dispatcher concentration.** `game_test.go`, `BotAssertionHandlers`, `BotStepCatalog.validate_step`, and `runtime_assertions.py` are the next high-leverage reads.
5. **CODEMAP now earns its keep.** The domain map routes future work to the new focused files instead of requiring broad coordinator reads first.

## 4. Top 10 recommendations

1. Split a cohesive `sim.go` gameplay domain with focused tests and replay proof.
2. Move touched `game_test.go` regions into domain test files instead of adding more cases there.
3. Split Python runtime assertions by domain, starting with movement/combat or inventory/shop/stash.
4. Split client bot assertion handlers by UI/gameplay domain.
5. Split `BotStepCatalog.validate_step` by step family before adding more scenario grammar.
6. Keep `main.gd` bot wrappers thin and continue extracting presentation domains only when touched.
7. Keep `run.py` as orchestration plus wrappers; new bot runtime behavior starts in focused modules.
8. Extract another validator from `validate_shared.py` when the next shared catalog grows.
9. Preserve market transactional audit discipline for future economy work.
10. Keep recording the grandfathered file total in reviews and expect it to move down.

## 5. What "done well" looks like here

The next batch should keep the same shape as v141-v149: small specs, focused module ownership, no behavior change for structural splits, targeted checks while iterating, full `make ci` before commit, CODEMAP updates when files move, and ratchet baselines lowered in the same slice as shrinkage. Feature slices can resume, but any slice that touches a grandfathered coordinator should leave it smaller or route new code into a focused file.
