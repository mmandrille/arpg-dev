## 3. Cross-cutting themes

1. **Feature work did not erode authority boundaries.** Recent client features display or request state; the server still owns combat, loot, inventory, objectives, and progression outcomes.
2. **Small extractions are working.** `interactables.go` is only 69 lines, but it removed a real behavior branch from `sim.go` and gave objective chest activation a focused home.
3. **The highest-risk files are stable but still too central.** The current line counts are `game_test.go` 9129, `sim.go` 7022, `main.gd` 6723, `run.py` 4294, `rules.go` 3268, and `validate_shared.py` 3169.
4. **The new extraction-coupling ratchet is useful.** It prevents future `helpers=globals()` style extraction claims from increasing and keeps the real typed-context refactor visible.
5. **Scenario coverage is strong enough that runtime cost matters.** Protocol bot coverage reached 110 scenarios; new proofs should stay narrow and avoid long travel/wait setup when direct selectors can prove the contract.

## 4. Top 10 recommendations

1. Extract dungeon runtime population from `sim.go`, starting with `populateDungeonLevel` and generated objective chest runtime IDs.
2. Drain touched `game_test.go` regions into focused domain test files instead of adding more generic gameplay tests there.
3. Extract inventory transfer/staging routing from `inventory_panel.gd` during the next inventory/equipment/market/stash slice.
4. Split `BotStepCatalog.validate_step` by scenario step family before adding more DSL validation.
5. Split `BotAssertionHandlers` by assertion domain before adding more client bot assertions.
6. Split `runtime_assertions.py` by protocol assertion domain, and avoid new helper-global forwarding.
7. Keep `tools/bot/run.py` as orchestration plus compatibility wrappers while moving new runtime behavior into focused modules.
8. Extract one catalog-specific validator from `validate_shared.py` when the next rules/protocol catalog grows.
9. Keep protocol bot scenarios under budget with direct setup, stable selectors, and narrow assertions.
10. Continue feature slices only when they keep gameplay values data-driven and leave touched coordinators no worse than they found them.

## 5. What "done well" looks like here

The next feature batch should pair visible gameplay slices with small ownership improvements. A good slice keeps server-owned outcomes authoritative, adds or updates one focused protocol bot proof, avoids hardcoding balance values in tests, updates SDD docs, and lowers any ratchet baseline it makes possible. If it touches a broad file, it should either extract the touched domain or explain why the extraction would be premature.
