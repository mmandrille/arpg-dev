## 1. Executive summary

The v161-v170 batch did what the v160 review asked for: it mixed feature closure with targeted paydown. Dungeon population moved out of `sim.go`, gold auto-pickup tests moved out of `game_test.go`, inventory routing moved toward a router, client bot UI assertions and action-step validation were split, protocol bot economy runtime assertions were split, and `main_config` validation moved out of `validate_shared.py`.

The architecture remains healthy. Server authority was preserved, the Godot client stayed display/input oriented, shared gameplay tuning remained data-driven, and full `make ci` passed at the v170 baseline. The next batch should keep paying down the same families, but the emphasis can shift from simple file splits to deeper coupling: typed context for bot runtime assertions, another `game_test.go` domain drain, and a first `main.gd` or `bot_scenario_runner.gd` ownership split.

## 3. Cross-cutting themes

1. **The paydown strategy is working.** The batch lowered large-file totals by 400 lines while shipping player/system-visible work.
2. **The next debt is coupling, not just size.** `runtime_assertions.py` still depends on helper dictionaries, and `tools/bot/run.py` remains the source of many runtime services.
3. **Client presentation stayed honest.** Inventory/session/objective UI work requests or renders state; it does not own outcomes.
4. **Focused helper modules are now the default pattern.** New helpers are small: `bot_ui_assertion_handlers.gd` 48 lines, `runtime_economy_assertions.py` 45 lines, `validate_main_config.py` 51 lines.
5. **Scenario coverage is broad enough to demand discipline.** CI now includes 111 protocol scenarios and 41 client bot scenarios; new scenarios should be narrow and budget-aware.

## 4. Top 10 recommendations

1. Drain another `game_test.go` domain into a focused test file, starting with interactables/range, door, teleporter, or boss-floor behavior.
2. Replace one `runtime_assertions.py` helper-global family with typed context plumbing.
3. Extract one `main.gd` presentation or panel-bridge domain during the next visible client feature.
4. Split another `validate_shared.py` domain, preferably shops, dungeon generation, or item presentations.
5. Keep future dungeon generation runtime additions in `dungeon_population.go`, not `sim.go`.
6. Move remaining pure equip-slot eligibility helpers from `inventory_panel.gd` into the transfer router when equipment routing is touched.
7. Extract action execution or state capture from `tools/bot/run.py` when a protocol scenario slice touches that area.
8. Split `bot_scenario_runner.gd` only when a client bot slice already touches scenario execution/state capture.
9. Keep all gameplay tuning in `shared/rules` or schema-backed content, with tests deriving expectations from rules.
10. Continue lowering maintainability baselines in the same slice as every shrink.

## 5. What "done well" looks like here

The next batch should keep the v161-v170 pattern: one small end-to-end proof at a time, server-owned outcomes, data-driven tuning, a focused Go/protocol/client proof as appropriate, and a meaningful ratchet improvement when a large file is touched. A strong v171-v180 batch would close at least one coupling issue, not only another line-count issue.
