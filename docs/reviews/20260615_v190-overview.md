## 3. Cross-cutting themes

1. **Server authority held.** New companions, affixes, set bonuses, and Paladin defenses are resolved by the Go sim and surfaced through protocol state/events.
2. **Rule ownership is good but not complete.** Skills and item rarities are data-driven, while companion follow/assist radii and monster-rarity expected values remain in Go implementation code.
3. **Presentation tracks protocol well, with one clear drift point.** Holy Shield visual radius reads shared rules, but Sanctuary dome radius is hardcoded in GDScript.
4. **Scenario coverage remains the project's best safety net.** The v181-v190 protocol scenarios and 45 client scenarios give strong regression confidence.
5. **Large coordinators are managed, not solved.** The ratchet is doing its job, but new slices still need local extraction before touching `main.gd`, `game_test.go`, `sim.go`, `rules.go`, `tools/bot/run.py`, or `validate_shared.py`.

## 4. Top 10 recommendations

1. Move companion AI follow/assist tuning into shared rules; derive Go tests and bot expectations from that data.
2. Make Sanctuary's dome radius read from `SkillRulesLoader`, matching the Holy Shield radius pattern.
3. Convert monster-rarity hardcoded expected weights/colors/scales in Go validation into a named golden/schema-owned check.
4. Extract the next touched test cluster from `server/internal/game/game_test.go`.
5. Split the next status-effect presentation sync path out of `client/scripts/main.gd`.
6. Extract the next itemization validator from `tools/validate_shared.py` when set items, rarity pools, or affixes are touched.
7. Keep all new bot assertion/action families out of `tools/bot/run.py` unless a typed-context split is part of the same task.
8. Preserve the improved `scripts/ci.sh` all-steps summary behavior; it is now valuable review evidence.
9. Use `$next` to choose a feature slice that can include one small local extraction before adding behavior.
10. Treat hired mercenaries from ADR-0010 as a separate product design step; do not blur them with current server-owned companion pets/revives without updating design docs.

## 5. What "done well" looks like here

- New gameplay remains server-authoritative and deterministic.
- Balance and AI feel values live in shared rules unless a spec explicitly justifies code ownership.
- Client visuals read protocol/shared-rule facts instead of copying tuning constants.
- Every feature slice adds focused bot or smoke proof without growing broad coordinators by default.
- Review and refactor cadence continues at roughly every 10 completed slices.
