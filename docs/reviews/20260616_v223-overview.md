## 3. Cross-cutting themes

1. **The architecture held under feature pressure.** The batch touched companions, economy/resources, upgrades, and unique effects without moving authority into the client.
2. **Client maintainability is trending in the right direction.** `main.gd` is still large but is smaller than at v200, with boss/blacksmith/companion helpers absorbing new behavior.
3. **Data ownership is still uneven.** Unique items and upgrade resources are data-driven; companion/elite movement tuning and monster-rarity validation still pin tunables in Go.
4. **CI is broad enough to catch regressions.** The full suite proved protocol, replay, client bots, and smoke; the transient store failure is worth watching but not blocking.
5. **Progress docs need link hygiene.** `PROGRESS.md` is compact and useful, but links inside `docs/progress/` still point as if they were at repo root.

## 4. Top 10 recommendations

1. Repair `docs/progress/*` Markdown links or add link validation to `scripts/check-progress-dashboard.sh`.
2. Move companion and elite-minion follow/assist tuning into shared rules with rule-derived tests.
3. Convert `validateMonsterRarities` hardcoded expected values into a named golden/schema-owned validation path.
4. Continue extracting the next touched presentation path out of `client/scripts/main.gd`.
5. Split one legacy `helpers=globals()` cluster in `tools/bot/run.py` into typed/narrow context and lower the coupling baseline.
6. Extract the next touched skill/class test cluster from `server/internal/game/game_test.go`.
7. Keep new resource/market persistence out of `server/internal/store/repos.go` unless paired with a focused split.
8. Continue splitting `tools/validate_shared.py` by moving the next rarity/item/unique validator into a focused module.
9. If `TestMarketReadSummaryExpiresListingsAndRefundsOffers` flakes again, make a focused deflake slice for market expiration cleanup/isolation.
