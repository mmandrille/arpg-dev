## 1. Executive summary

The v232 baseline is healthy. The selected `$autoloop` queue shipped dungeon ambience audio, skill
audio cues, resource wallet presentation, material auto-pickup, set/unique mystery eligibility,
market cancellation UI, and mercenary recovery UI. The final batch `make ci` passed after v232,
and current review checks passed shared validation, Go tests/vet, Python tests, client unit tests,
and maintainability ratchets.

Architecture boundaries held: the client added presentation, audio, and intent surfaces, while the
server still owns resource pickup, item/economy outcomes, market cancellation, mercenary state, and
session persistence. The most meaningful improvement since v223 is process/tooling paydown:
progress-link validation now exists and helper-global coupling dropped from 37 to 31 calls.

The main current risks are not immediate correctness blockers. Companion and elite-minion movement
tuning still needs shared-rule ownership, monster-rarity validation still pins tunables in Go, and
large client/backend/tool coordinators remain managed debt. There is also a small process drift:
agent entrypoint docs disagree on whether `$review` or `$refactor` runs first.

## 3. Cross-cutting themes

1. **The thin-client/server-authoritative split is still intact.** New UI and audio features render
   or emit intents; server/store paths own actual item, resource, market, and mercenary outcomes.
2. **Prior process debt was paid down.** Progress archive link validation is now part of
   maintainability, and helper-global coupling dropped materially.
3. **Data ownership remains the highest-value architecture follow-up.** Companion/elite tuning and
   monster-rarity expectations still live in code where rules/goldens would be healthier.
4. **Large files are stable but still expensive.** The ratchet prevents regression, but new work
   should keep bypassing or extracting from `main.gd`, `market_panel.gd`, `game_test.go`,
   `repos.go`, and `validate_shared.py`.
5. **Verification is broad, with one command caveat.** Standalone `make client-smoke` failed because
   no server was listening; `make ci` owns that harness and was already green after v232.

## 4. Top 10 recommendations

1. Align agent docs so they consistently say `$review` writes the fresh scorecard before
   `$refactor` classifies and pays down recommendations.
2. Move companion and elite-minion follow/assist tuning into shared rules with current-value parity
   and rule-derived tests.
3. Convert `validateMonsterRarities` hardcoded expected values into a named golden or schema-owned
   validation path.
4. Replace the next `helpers=globals()` wait/runtime cluster in `tools/bot/run.py` with typed
   context or direct helper imports, then lower the coupling baseline.
5. Continue extracting the next touched `main.gd` routing path, prioritizing bot facade, entity
   reaction, or projectile presentation.
6. Split future market UI work out of `client/scripts/market_panel.gd`.
7. Split the next touched skill/class assertion cluster out of `server/internal/game/game_test.go`.
8. Continue extracting focused validators from `tools/validate_shared.py` when rarity/item/unique
   checks are next touched.
9. Clarify standalone `make client-smoke` server prerequisites if agents keep running it outside
   `make ci`.
10. Keep next feature proposals sourced from review follow-ups or active open gaps, with data-driven
    ownership called out in specs.
