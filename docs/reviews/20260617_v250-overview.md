## 1. Executive summary

The v250 baseline is healthy. The selected `$autoloop` queue shipped market expiration/search/sort/
comparison UI, material wallet presentation, blacksmith recipe/history UI, mercenary combat stat
display, ranged companion projectile targeting, boss reward UI, and boss-specific telegraph decals.
The final batch `make ci` passed after v250 in 10m27s, including the full Godot client bot suite
with 66 passing scenarios.

Architecture boundaries held. The server still owns market listing status/expiration, blacksmith
upgrade validation, companion combat stats, ranged monster projectile targets, damage, loot,
inventory, and persistence. The Godot client renders state, emits existing requests/intents, and
uses presentation-only helpers for market rows, boss rewards, and telegraph decals.

The highest-value improvements since v232 are process and maintainability trend fixes: agent docs
now agree on `$review` before `$refactor`, `client-smoke` prerequisites are explicit, companion
follow tuning is data-driven through `main_config`, the old monster-rarity hardcoding finding is no
longer supported by current code, and helper-global coupling dropped from 31 to 14 calls.

Remaining risks are managed debt rather than correctness blockers: v241-v250 proof docs still say
batch CI is pending, client debug progression JSON seeding needs a focused unit proof, `tools/bot/run.py`
still owns 14 helper-global injections, and large coordinators remain under the file-size ratchet.

## 3. Cross-cutting themes

1. **The server/client authority split remains intact.** The client added richer market, blacksmith,
   mercenary, and boss presentation, while authoritative outcomes stayed in Go/store/protocol paths.
2. **Prior review drift has mostly been paid down.** Review/refactor order is aligned across agent
   entry docs, `client-smoke` prerequisites are explicit, companion config moved into shared rules,
   and helper-global debt shrank to 14.
3. **Documentation proof needs one cleanup pass.** The final CI is green, but v241-v250 lifecycle and
   as-built files still describe the batch as pending.
4. **The next test gap is small and concrete.** Client bot debug progression JSON now supports full
   nested objects, but the integer coercion/parsing helper is only protected indirectly by live boss
   scenarios.
5. **Large-file pressure remains managed debt.** Ratchets prevent regression, but future slices
   should keep extracting or bypassing `main.gd`, `game_test.go`, `run.py`, `validate_shared.py`,
   `market_panel.gd`, `shop_panel.gd`, and large client tests.

## 4. Top 10 recommendations

1. Update v241-v250 lifecycle/as-built proof wording so it records the passed batch `make ci`.
2. Add a focused GDScript unit for client debug progression JSON parsing and `NetClient._int_value_map`.
3. Replace the next `tools/bot/run.py` helper-global cluster with typed context/direct helper calls,
   then lower `.maintainability/extraction-coupling-baseline.tsv`.
4. Split the next touched `main.gd` presentation/debug-row path rather than adding more inline keys.
5. Split or share future market/shop comparison display helpers when either panel is next touched.
6. Split the next touched backend assertion cluster out of `server/internal/game/game_test.go`.
7. Split the next touched validator domain out of `tools/validate_shared.py`.
8. Plan separate shared-rule naming for elite-minion follow tuning when AI movement tuning is next
   in scope.
9. Continue the tuning-friendly test audit from `PROGRESS.md`: convert accidental balance pins to
   rule-derived, semantic, range, or eventual assertions while preserving intentional goldens.
10. Keep production boss telegraph VFX, exact projected aim decals, and additional boss patterns as
    future planned slices rather than incidental refactors.
