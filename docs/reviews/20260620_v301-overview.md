## 1. Executive summary

The v301 baseline is stronger than v284. The selected Movement / Combat Fluidity batch shipped
seven player-visible feel improvements: input buffering, attack move / sticky targeting, immediate
local windup/recovery presentation, hit stop / impact flash, movement visual smoothing, command
retarget grace, and melee lunge. The implementation stayed on the intended side of the authority
boundary: helpers are client-only and continue to use existing server-owned `action_intent` and
`move_to_intent` outcomes.

The v284 broad client-bot flakes are no longer the headline risk. The final post-loop sweep reported
85 client scenarios passed / 0 failed, and the mercenary recovery failure was fixed with a concrete
stale-companion cleanup and unit proof. The previous model-catalog drift recommendation is also
materially resolved by a generated shared model preview catalog plus pytest coverage.

The main review finding is infrastructure/process: raw `make ci` cannot currently complete from an
isolated worktree when the main checkout already owns the fixed local `arpg-postgres` container. The
batch was verified with a CI-equivalent continuation against the existing healthy DB and temporary
servers, but the next `$refactor` should make the raw gate worktree-safe.

## 3. Cross-cutting themes

1. **Client feel improved without server-authority drift.** Local helpers own presentation and input
   timing only; combat and movement outcomes still come from the existing server protocol.
2. **Bot proof quality improved.** The client scenario count rose to 85, and the full sweep passed
   after stabilizing command retargeting and mercenary recovery.
3. **The model-viewer catalog debt is mostly closed.** Python generates the shared catalog and Godot
   consumes it, with pytest drift detection.
4. **Raw CI is too coupled to one local Docker container name.** Worktree isolation exposed a real
   process gap even though the code gates passed through equivalent continuation.
5. **Feel tuning needs a shared home before it grows.** The v299-v301 specs documented code-owned
   microconstants, but a central presentation-feel config would better match the repo policy.

## 4. Top 10 recommendations

1. Make `make ci` database/server bootstrap worktree-safe so isolated worktrees can run the raw
   full gate without conflicting with a healthy main-worktree `arpg-postgres`.
2. Add a shared presentation-feel config or central code-owner module for client-only input/presentation
   constants before more movement/combat tuning spreads across helpers.
3. Extract the next touched movement/combat input orchestration out of `client/scripts/main.gd`.
4. Split the next touched `client/scripts/bot_scenario_runner.gd` step family into handler/test files.
5. Split the next touched backend assertion cluster out of `server/internal/game/game_test.go`.
6. Split the next touched validation domain out of `tools/validate_shared.py`.
7. Keep model preview catalog parity checks green; consider adding a named `--check` target if the
   model preview surface expands.
8. Resolve or replace `user-provided-unverified` class GLB provenance before production distribution.
9. Continue the tuning-friendly rule-test audit so balance/rule changes do not require unrelated
   exact-value edits.
10. Keep production combat VFX/audio/art polish as feature slices with visual bot proof, not refactor
    opportunism.

## 5. What "done well" looks like here

- `$refactor` first makes the local CI server bootstrap robust enough that raw `make ci` can run
  from the isolated worktree or clearly reuse a healthy DB.
- Future movement/combat feel tuning has one data-backed or explicitly documented owner.
- `main.gd`, `bot_scenario_runner.gd`, `game_test.go`, and `validate_shared.py` shrink when next
  touched rather than accumulating more behavior.
- Asset provenance is clear enough that shipping/distribution decisions do not require local-history
  forensics.
