## 1. Executive summary

The v309–v331 run delivered 23 slices of meaningful content: passive skill columns, room-based dungeon
layout, movement-speed extraction, hero visibility / lighting / fog, minimap discovery, presentation
polish, and numerous bug-fix stabilization passes. Several v308 refactor targets landed cleanly — the
dead `obstacleBlocksMovement` switch is gone, `dungeon_floor_features.go` unified the water/hole
duplication, shop views were extracted from `rules.go`, the dungeon-obstacles `cross_checks()` guard was
added, the CLAUDE.md rule #12 / run.py budget contradiction was reconciled, and all five v308 orphan
unit tests are now wired in `client_smoke.sh`. These are genuine improvements that compound.

The period ends with **one hard blocker and two persistent structural debts.**

The blocker: **`go test ./...` is red at HEAD** — 7 failures in the `game` package and 1 in `replay`,
covering the pickup/auto-pickup/door family, two collision tests, a dungeon obstacle clearance
violation, and a replay golden mismatch. Nothing in this review matters until the test suite is green.

The first structural debt is **`docs/CODEMAP.md`**, which this review flags for the **third consecutive
time**. The v309–v331 run added at least 7 new production files; none appear in the index.
`validate_codemap.py` still cannot detect unlisted present files. The next agent that loads the CODEMAP
as its "focused context" will miss every new domain file from the last three batches. This is
cumulative damage to the SDD workflow.

The second structural debt is **the two largest non-test files: `sim.go` (6,541 LOC) and `rules.go`
(3,308 LOC)**. Both are grandfathered and received no net reduction over 23 slices despite multiple
extraction opportunities. On the client, `main.gd` reached 6,084 lines without any baseline lowered.
The maintainability ratchet is recording the debt correctly; it is not being paid down.

Architecture discipline and the data-driven rules boundary remain the repo's clearest strength — every
new mechanic is server-authoritative, every new tuning value is in `shared/rules/`, and the
determinism linter is clean.

## 3. Cross-cutting themes

1. **The test suite must be green before any new feature slice.** The 7+1 failures touch core
   gameplay mechanics (pickup, collision, door, dungeon generation, replay determinism). They imply
   that at least one recent commit introduced a regression that was not caught before landing on main.
   The CI gate policy exists for exactly this reason.

2. **`docs/CODEMAP.md` staleness is compounding.** Three consecutive reviews have flagged this. The
   fix is two parts: (a) update the index for v302–v331 new files, and (b) add a single inverse check
   to `validate_codemap.py` so CI fails when a `server/internal/game/*.go` or
   `client/scripts/*.gd` file exists but is absent from the index. Until (b) lands, the index will
   rot again with the next batch.

3. **The three largest files are structural debt that compounds every slice.** `sim.go` (6,541),
   `main.gd` (6,084), and `rules.go` (3,308) absorb new functionality because they are already the
   primary homes for their domains. New focused files appear alongside them correctly, but the
   coordinators themselves do not shrink. The ratchet records the problem; it does not solve it.

4. **v308 minor-refactor recommendations largely landed.** Dead switch, water/hole duplication,
   cross-checks, budget contradiction, orphan test wiring — all resolved. The process works when the
   fixes make it into the `$refactor` queue. The outstanding item (CODEMAP) is the one that was
   explicitly labeled "most cited" and yet is still open.

5. **New schema in shared/assets has gaps.** `fog_presentation.v0.schema.json`'s `point_light` block
   has `additionalProperties: false` but no `required` entries — a missing field produces undefined
   GDScript behavior rather than a schema validation failure. The `height_fraction` and `min_height`
   fog keys added in v331 are also unvalidated by `validate_shared.py`.

## 4. Top 10 recommendations

Tags: **[blocker]** = must land before next feature slice; **[minor]** = safe `$refactor` commit;
**[future]** = feature slice or design work outside `$refactor` scope.

1. **[blocker · Tech] Fix the 7+1 failing Go tests.** The `game` package failures (pickup, collision,
   door, obstacle clearance) and the `replay` golden mismatch must be resolved and CI must be green
   before any new feature work. Identify the root-cause commit with `git bisect` if not obvious.

2. **[minor · Docs+Process] Update `docs/CODEMAP.md` + add inverse check to `validate_codemap.py`.**
   Add all new server/client/shared files from v302–v331 to the index. Strengthen
   `validate_codemap.py` to fail when any `server/internal/game/*.go` (non-test) or
   `client/scripts/*.gd` file is absent from the listed entries. This closes the rot class
   permanently. Third consecutive review flagging this — escalated to second priority.

3. **[minor · Schema] Add `required` fields to `fog_presentation.v0.schema.json`'s `point_light`
   block.** `additionalProperties: false` without `required` means a missing `shadow_enabled`,
   `energy`, or `range` silently omits the field and produces undefined GDScript behavior. Add a
   `validate_shared.py` semantic range guard for `falloff_power` and `energy` to catch
   black-screening tuning errors before they hit the client.

4. **[minor · Maint] Add `fog_of_war_overlay.gd` to the maintainability baseline.** The file crossed
   600 lines in v331 without a baseline entry — it will fail the ratchet CI gate if committed as-is.
   Either add the entry now (with a plan to extract `OmniLight3D` management into a focused helper)
   or extract the ~60-line light node lifecycle before closing the slice.

5. **[minor · Maint] Extract type clusters from `rules.go` into domain files.** 80 type definitions
   live in `rules.go` (3,308 lines). A `dungeon_generation_rules.go`, `passive_skill_rules.go`, and
   `shop_rules.go` split would lower the baseline toward the 600-line target and make each domain
   independently loadable for focused reviews and tests.

6. **[minor · Process] Document the v322–v328 spec-exemption pattern.** Pure VFX/presentation
   slices that touch no protocol, server state, or shared rules are reasonable to ship without a
   formal spec. Record that exception explicitly in CLAUDE.md (e.g., "presentation-only slices that
   touch no protocol or server state may skip the spec gate") so the deviation is intentional, not
   invisible.

7. **[minor · Maint] Begin extracting the movement-intent orchestration cluster from `main.gd`.** The
   file is at 6,084 lines. The movement-intent cluster (input → intent → visual feedback path) is
   a well-bounded seam that does not require `main.gd`'s full scene context. Extract it to a focused
   `movement_input_presenter.gd` and lower `main.gd`'s baseline.

8. **[minor · Docs] Write an ADR (or ADR-0001 addendum) for the movement-speed formula boundary.**
   The formula (class base × DEX multiplier × gear percent) was established in v332 and now governs
   replay determinism, bot timing budgets, and the client's prediction model. Recording it as an ADR
   prevents future contributors from unknowingly breaking the cross-language contract.

9. **[minor · Test] Add a semantic range `cross_checks()` guard in `validate_shared.py`** for the
   new fog presentation tuning fields (`falloff_power`, `intensity`, `energy`, `range` in
   `fog_presentation.v0.json`) to catch values that pass schema but produce black-screen or
   invisible-fog rendering bugs.

10. **[future] Continue the coordinator extraction program.** `sim.go` (6,541 lines) is the single
    largest file in the repo. A typed tick-context struct passed through handlers would enable
    incremental extraction of the per-tick update logic into focused files without breaking the
    determinism guarantees. This is a multi-slice architectural investment, not a `$refactor` one-shot.

  lower baselines than v331, not equal ones.
- The data-driven / server-authority discipline that made the v302–v308 batch strong continues as
  the default — every new mechanic and every new tuning value follows the same pattern.
