## Summary

The Go server is the strongest layer in the repo on *correctness* and the weakest on *internal decomposition*. The authoritative boundary is clean (the `game/` package imports no WebSocket, DB, or HTTP), determinism is genuinely enforced through a custom RNG and pervasive sorted-iteration helpers, and the DB layer is parameterized and transaction-safe. The problem is `sim.go`: a **7,036-line god object** with ~16 distinct responsibilities and no internal seam, mirrored by a **7,144-line `game_test.go`**. Every new slice edits the core file. That is the dominant maintainability risk.

**Backend sub-scores:** Architecture-boundaries 9.0 · Architecture-cohesion 5.5 · Technical 8.0 · Maintainability 6.0 · Documentation 6.5.

## 1. Architecture

- **`sim.go` is a god object. [High]** 7,036 lines, ~329 methods on `*Sim`. At least 16 concerns live here: entity lifecycle, ~24 intent handlers, combat resolution, projectile physics, monster AI, pathfinding coordination, dungeon hydration, loot dispatch, shop/stash, inventory/hotbar/equip, stat derivation, XP/leveling, skill cooldowns, snapshot serialization, replay entry points, math utils. `dungeon_gen.go` (909) and `shop.go` (1,110) show the right extraction instinct — but `sim.go` still owns their call sites and everything else.
- **Authoritative boundary is clean. [Strength]** `game/` imports only stdlib (`encoding/json`, `fmt`, `math`, `sort`, `strconv`). `realtime/`→`game/`, `replay/`→`game/`, `http/`→`game/` (types + `Rules`) all flow the correct direction. The sim has no knowledge of sockets, DB, or HTTP. This is what makes replay determinism and the thin-client contract actually hold.
- **One concurrency crack. [Med]** `session_loop.go:502` calls `l.sim.PlayerCurrentLevel()` inside `fanoutResult`, which runs **after** `l.mu` is released (`session_loop.go:488`). The `clients` slice is snapshot-copied under lock, but `l.sim` itself is read unlocked. Safe today because the tick loop is single-goroutine; a latent data race the moment any background/reconnect path touches the sim concurrently.
- **Tick loop is well-built. [Strength]** `session_loop.go:447` — `time.NewTicker`, inputs `sortInputs`'d before delivery, mutex scoped to sim+buffer only, I/O (persist, fanout) runs unlocked.
- **There is exactly one live wire format.** The 9 on-disk schema versions (`shared/protocol/*.v0–v8`) are additive history; server decode (`inputdecode`) is a flat version-agnostic type switch. No runtime version negotiation exists (fine for a single-client-version project, but worth stating in an as-built doc).

## 2. Technical

- **Determinism — mostly enforced, 2 real violations. [Med]**
  - Clean: zero `time.Now()` in `game/`; no `math/rand`; custom splitmix64 in `rng.go`; `sorted*` helpers used pervasively (e.g. `sim.go:974,1076,1137,2018,2876`).
  - **Violation:** `sim.go:3150` (`handleDrop`) — `for slot, instanceID := range s.equipped` (map). The early `break` on match usually masks it, but the position of the emitted `OpEquippedUpdate` in `res.Changes` is not guaranteed stable. **Med.**
  - **Violation:** `sim.go:5382` (`equipPreviewForItemWithSlot`) — same bare range; read-only, so lower risk. **Low.**
  - `rules.go` map ranges (`865,947,1099,1478`) are inside `LoadRules` (startup validation) → acceptable.
- **`panic(err)` in `NewSim`. [High]** `sim.go:298` panics the process if `NewSimWithWorld` fails for the default world. A rules misconfiguration kills the server instead of returning an error the HTTP/realtime layer can handle. The error-returning variants already exist downstream — `NewSim` should propagate, not panic. (`sim.go:1556` `panic("active level missing")` is an acceptable invariant guard.)
- **DB layer is solid, one N+1. [Med]** `repos.go` (1,871 LOC): all queries use `$N` params (no injection surface); transactions via `pgx.BeginFunc` (`165,384,475,1232`). But `LoadSessionStartSnapshots` (~`1619`) loops members calling `LoadSessionStartSnapshotForMember`, each firing 5–7 queries → ~28 round-trips for a 4-player co-op join. Only at join time, not hot-path, but worth batching with `ANY(...)`.
- **Migrations:** 14 sequential numbered SQL files, self-applied on boot. Clean, framework-free.
- **Hygiene:** `go vet ./...` clean. **Zero** TODO/FIXME/HACK markers in `game/` or `store/`. No ignored errors in `game/`. Lean deps.

## 3. Maintainability

- **No seam for new slices. [High]** `applyInput` (`sim.go:1454`) is a 59-line if-chain routing to ~24 `handle*` methods, all in `sim.go`. Adding an intent means editing `sim.go` + `inputdecode/inputdecode.go` + `realtime/protocol.go`. There is no dispatch table, handler registry, or interface boundary.
- **`game_test.go` (7,144 lines) is "everything not already split out".** Shop/stash/replay tests live in their own files; the remainder is one monolith navigable only by grep. Grows monotonically per slice. Uses a `loadGolden` helper (`game_test.go:39`).
- **Naming is consistent. [Strength]** `handle*` (inputs), `sorted*` (deterministic iteration), `*View` (protocol structs), `*State` (mutable). No drift found.
- **Minor archaeology:** the `// v0 slice` constants block (`sim.go:11`) is stale at 53 slices; `monsterDefID = "training_dummy"` is now world-config-driven.

## 4. Documentation

- **No package doc on `game/`. [Med]** None of `sim.go`, `rules.go`, `types.go`, `dungeon_gen.go`, `pathfind.go` carry a `// Package game` comment. `realtime` and `inputdecode` do.
- **Type docs are excellent. [Strength]** `types.go` documents nearly every exported struct/field; `rng.go` documents the splitmix64 rationale with ADR reference.
- **`dungeon_gen.go` is effectively undocumented. [High]** 909 LOC / 47 functions / **2 comment lines**. The room placement, corridor logic, and champion/minion distribution — the most complex algorithmic code in the repo — have no inline explanation. `pathfind.go` (197 LOC, 1 comment) is nearly as opaque.
- **Determinism invariant is doc-only, not gated. [Med]** Enforced by CLAUDE.md prose + review, not by any lint. The two bare-map-range violations are proof the gate is needed.
- **Doc/code drift:** `protocol.go:2` package doc says "20 Hz"; the live loop is **15 Hz** (`protocol.go:17`, with a clarifying comment at `:14`). `CLAUDE.md` also says "fixed 20 Hz tick." The sim still advances one fixed tick/loop, so replay is unaffected — but the stated rate is wrong in two reader-facing places.

## Top 5 backend refactors

1. **Extract an input-handler registry** (`sim.go:1454` + 24 `handle*`): `map[InputType]func(*Sim, Input, *TickResult)` in `handlers.go`. New slices register instead of editing the dispatcher — removes the #1 reason to touch `sim.go`. *(med/high)*
2. **Fix the two bare `s.equipped` ranges** (`sim.go:3150`, `:5382`) → iterate `sortedStringKeys(s.equipped)`. Closes the last hot-path determinism hole. *(trivial/high)*
3. **Add a CI determinism lint** (analysistest or `staticcheck` custom): fail on `time.Now()`, non-seeded rand, and bare map range in `game/`. Convert the core invariant to a gate. *(med/high)*
4. **Replace `panic(err)` in `NewSim`** (`sim.go:298`) with error return + propagation. *(low/med)*
5. **Document `dungeon_gen.go` and `pathfind.go`**, and add a `// Package game` doc stating the determinism contract. *(low/med)* — and batch `LoadSessionStartSnapshots` to kill the join-time N+1. *(med/med)*
