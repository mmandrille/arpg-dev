## Summary

The backend is healthier than it was at v53. The v55 handler registry moved input dispatch out of
the old `sim.go` chain, and the determinism lint is now a CI gate. The server still has the
project's strongest correctness posture: rules load once, the sim owns outcomes, replay remains
deterministic, and validation rejects most data drift before runtime.

The remaining backend risks are narrower but still worth prioritizing. `NewSim` still panics on
startup rule/world errors, `sessionLoop.fanoutResult` still reads sim state after the loop mutex is
released, and `rules.go` is becoming the next content-loading monolith exactly as skills/items move
toward library manifests.

## 1. Architecture

- **[Strength] Input dispatch now has a registry seam.** `server/internal/game/handlers.go:24`
  defines `inputHandlers`, and `server/internal/game/sim.go:1455` only performs registry lookup.
  This resolves the v53 "new intent edits core dispatcher" finding.
- **[Strength] Determinism is now enforced by tooling.** `make/server.mk:12` runs
  `determinism-lint`; `scripts/ci.sh:32` includes it as CI step 3/9; `make lint-determinism`
  passed during this review.
- **[High] Default sim construction can still crash the process.** `server/internal/game/sim.go:296`
  returns `*Sim`, calls `NewSimWithWorld`, and `panic(err)`s at `server/internal/game/sim.go:299`.
  The error-returning constructor already exists, so startup callers should stop using the panic
  wrapper for any path fed by external data.
- **[Med] Realtime fanout reads sim state after unlocking.** `server/internal/realtime/session_loop.go:487`
  unlocks `l.mu`, then `fanoutResult` reads `l.sim.PlayerCurrentLevel` at
  `server/internal/realtime/session_loop.go:502`. The single tick goroutine makes this safe today,
  but reconnect/background reads would turn it into a race. Snapshot client levels while locked.

## 2. Technical

- **[Strength] Hot-path map range hazards from v53 are closed.** The old `s.equipped` bare ranges
  now iterate through `sortedStringKeys` in `server/internal/game/handlers.go:745` and
  `server/internal/game/sim.go:4350`.
- **[Med] `rules.go` is the next loading bottleneck.** `LoadRules` starts at
  `server/internal/game/rules.go:716`, validates combat/progression/world/boss/shop/skill data, and
  now also owns generic skill checks. A skills-only content manifest should extract loader helpers
  rather than grow this function further.
- **[Strength] Skill catalog validation is real server-side authority.** `shared/rules/skills.v0.schema.json:16`
  requires every supported gameplay field, and `tools/validate_shared.py:1061` checks presentation
  coverage against skill IDs. The next manifest slice can reuse these checks without behavior
  change.
- **[Low] `game_test.go` remains the backend test sink.** At 7,350 LOC it still absorbs broad sim
  coverage. New content-library tests should go into focused loader/test files to avoid regrowing
  the old monolith pattern.

## 3. Maintainability

- **[Strength] v55 paid down the most urgent backend god-file debt.** `sim.go` is 6,029 LOC now,
  down from the v53 review's 7,036 LOC, and handler code has a dedicated file.
- **[Med] Content loading has no manifest abstraction yet.** The v59 research note correctly frames
  file paths as indexes, not gameplay IDs. A skills-only manifest is a good next step because the
  current skill catalog is small, schema-backed, and has one client loader.
- **[Med] Startup constructors have two styles.** Tests and infra call the error-returning
  constructors heavily, but the panic wrapper remains exported. Standardizing on error returns
  would align runtime failure behavior with the rest of `LoadRules`.

## 4. Documentation

- **[Strength] Agent-facing docs now match current invariants.** `CLAUDE.md:99` correctly states a
  15 Hz tick and documents `make lint-determinism`.
- **[Low] Package docs are still thin.** `server/internal/game/sim.go:1` starts at `package game`
  without a package-level determinism contract. The current behavior is documented elsewhere, but
  the package itself should carry the invariant.
- **[Med] Manifest rollout needs an ADR or as-built decision once implemented.** The research note
  is enough to guide the first slice, but content library loading will become an architectural
  rule across skills, items, classes, and assets.

## Top 5 backend refactors

1. Replace `NewSim` panic usage with error-returning construction on runtime paths.
2. Snapshot per-client levels while holding `sessionLoop`'s mutex before fanout.
3. Extract skill/content loader helpers from `rules.go` before adding a manifest.
4. Move new skill catalog tests into focused loader tests, not `game_test.go`.
5. Add a package doc for `game` that states the deterministic sim contract and loader boundary.

`make/server.mk`, `scripts/ci.sh`; `make lint-determinism` passed.*
