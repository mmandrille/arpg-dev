## Summary

The backend remains structurally sound. Combat, aggro, loot, dungeon generation, inventory, and replayable outcomes are still server-owned, and recent pack work made generated encounters more intentional without widening the protocol surface. The main backend risk is a bounded realtime correctness issue that survived the v70 review: fanout still reads sim state after the loop mutex is released.

Checks run for this refresh: `make validate-shared` passed with 720 checks, and `cd server && go test ./... && go vet ./...` passed.

## 1. Architecture

[Strength] Dungeon packs are generated from explicit rule data and remain internal until a gameplay or UI feature needs to expose them. Pack sizing and composition happen in `server/internal/game/dungeon_gen.go:609` and `server/internal/game/dungeon_gen.go:642`; leader handling stays inside generation at `server/internal/game/dungeon_gen.go:772`.

[Med] Realtime fanout still reads player current levels after the session loop mutex is released. `session_loop.go` copies clients under lock at `server/internal/realtime/session_loop.go:508`, unlocks at `server/internal/realtime/session_loop.go:514`, then calls `l.sim.PlayerCurrentLevel` inside fanout at `server/internal/realtime/session_loop.go:529`. If session membership or level state changes concurrently, fanout can observe a state different from the tick result being persisted. This was noted in v70 and remains the highest-value backend maintenance slice.

## 2. Technical

[Strength] Monster behavior validation catches impossible rule combinations before runtime. Static monsters reject aggro/assist/leash/move/attack fields at `server/internal/game/rules.go:1373`, while chase monsters validate aggro, assist, leash, and movement bounds at `server/internal/game/rules.go:1383`.

[Strength] CI shape still protects determinism and replayability. `scripts/ci.sh:150` runs protocol bots plus replay, `scripts/ci.sh:176` runs Godot client bot scenarios, and `scripts/ci.sh:184` runs headless smoke as the final local CI phase.

[Low] Elite leader metadata is currently internal-only, which is appropriate. Future tests should keep using semantic assertions around pack roles and leader presence rather than exact total monster counts, because champion minions and pack sizing are legitimate data-driven outputs.

## 3. Maintainability

[Med] `server/internal/game/game_test.go` is 8247 lines. It provides valuable coverage, but new tests for generation, combat, shops, skills, and persistence should land in focused files when those areas are touched.

[Low] `server/internal/game/rules.go` is 2853 lines and owns many content-domain validation rules. This is still effective, but future rule domains should prefer small validation helpers or files rather than extending the central loader indefinitely.

## 4. Documentation

[Strength] The backend behavior remains well anchored by `CLAUDE.md` invariants: new intents register in `handlers.go`, sim determinism is linted, and shared rules remain data. The current review evidence confirms those process rules still match the repository shape.

## Top 5 backend refactors

1. Snapshot each client player's level under the session loop mutex before `fanoutResult`.
2. Split new dungeon generation tests into `dungeon_gen_test.go`.
3. Add a short package doc for `server/internal/game` explaining deterministic boundaries and seeded RNG ownership.
4. Keep pack leader metadata internal until a concrete UI/gameplay feature needs it on the wire.
5. Continue replacing exact generated population assertions with rule-derived or semantic assertions.
