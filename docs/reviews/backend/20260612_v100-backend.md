## Summary

The backend remains architecturally sound: the Go sim owns gameplay outcomes, shared JSON owns tuning/content, and protocol events are the presentation bridge. The v90-v99 batch added class/loadout/rogue combat work without breaking that boundary.

The next requested work, damage types and resistances, is a good fit for the current architecture but should be introduced as data, not scattered conditionals. It will touch the combat formula path, monster definitions, item/skill rules, bot proof, and likely shared validation.

## 1. Architecture

[Strength] Server authority is still explicit in code and docs. The client declares itself as a thin renderer at `client/scripts/main.gd:1`, while server combat data and rules load through `server/internal/game/rules.go:13`.

[Med] Combat behavior is still concentrated in `server/internal/game/sim.go:1`, a 7566-line file. Damage-type application should land behind a focused helper/file so future poison, cold, lightning, fire, and force rules do not make the sim monolith harder to reason about.

[Med] Test coverage exists but `server/internal/game/game_test.go:1` is now 8766 lines. New combat tests for resistance math should live in a focused test file rather than extending the central fixture file.

## 2. Technical

[Strength] The deterministic constraints remain visible: sim code imports no wall clock at `server/internal/game/sim.go:3`, and repo CI still includes determinism lint through the existing `make ci` path.

[Med] Damage typing is not yet a first-class contract. The current rules expose raw damage ranges in item/monster/skill data, but not a shared damage-type enum or resistance map. The next slice should add validation before gameplay code depends on the fields.

[Low] The existing spelling drift around `ligthing`/`lightning` needs deliberate handling in the upcoming slice. Use `lightning` for new canonical ids unless existing content already forces a compatibility alias.

## 3. Maintainability

[Med] `server/internal/game/rules.go:16` keeps accumulating rule domains. Adding damage types and resistances should either use a small validation helper or clearly isolate the new validation block.

[Low] New monster families should stay data-defined. The requested undead enemy should not add special-case server code for poison immunity if the resistance system can express `poison: 1.0`.

## 4. Documentation

[Strength] Specs, plans, and as-built docs are current through v99 in `PROGRESS.md`.

[Med] The damage-type slice should document formula semantics precisely: resistance fraction, weakness as negative resistance, rounding, and event payload expectations.

## Top 5 backend refactors

1. Extract damage application into a focused combat helper before adding per-type resistance rules.
2. Put damage-type and resistance tests in a new focused Go test file.
3. Add shared validation for canonical damage types and resistance ranges.
4. Keep undead poison immunity as data in monster rules.
5. Preserve replay determinism by deriving all new calculations from existing sim inputs and shared rules.
