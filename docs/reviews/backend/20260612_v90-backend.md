## Summary

The backend continues to preserve the core architecture: the server owns gameplay facts, validates shared rules, emits replayable state/events, and lets the client render presentation. Recent skill work stayed data-driven enough that generic tooling can replay new skills without dedicated scenarios.

The key v80 backend correctness finding is resolved. Realtime fanout now snapshots player levels while holding the session loop mutex, then uses that snapshot during result fanout.

## 1. Architecture

[Strength] Server authority remains intact for combat and skills. New skills continue to flow through shared skill data, server-side rank/cooldown/effect handling, and protocol events rather than client-side gameplay shortcuts.

[Strength] The realtime fanout level race identified in v70/v80 is closed. `server/internal/realtime/session_loop.go` captures `levelsByPlayerID` under the session loop mutex and passes the snapshot to fanout, so outbound deltas use tick-time level membership.

[Med] The deterministic sim is still too concentrated in `server/internal/game/sim.go` at 7207 lines. It works, but future skill, monster, and status-effect changes should prefer focused files/helpers around the touched domain.

## 2. Technical

[Strength] Full CI is green after blocker cleanup, including Go test/vet, shared validation, protocol bot replay, client bots, and headless smoke.

[Low] The review cleanup changed the skill XP dummy used by skill-lab proofing to 1 HP. That is appropriate for scenario determinism, but similar future proof monsters should stay explicitly marked as test/lab content in shared data so they cannot be mistaken for balance targets.

## 3. Maintainability

[Med] `server/internal/game/game_test.go` remains the largest repo file at 8619 lines. Add new tests in focused files when touching skills, monsters, shops, generation, persistence, or replay rather than extending the central test file.

[Low] `server/internal/game/rules.go` is still a broad validation owner. It is acceptable for now, but new content domains should extract domain validators or at least local helper blocks.

## Top backend recommendations

1. Split new server tests by domain instead of extending `game_test.go`.
2. Extract touched sim domains when adding new status effects, skill capabilities, or combat formulas.
3. Keep proof monsters clearly separated from production balance data.
4. Add localization text-key validation in shared rule loading once text keys are introduced.
5. Preserve the current server-owned skill/event boundary while localization work changes presentation text.

*Evidence: direct reads of `server/internal/game`, `server/internal/realtime`, shared rules, `scripts/ci.sh`, and successful verification listed in the overview.*
