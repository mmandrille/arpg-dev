# arpg-dev - Backend Review at slice **v110**

**Date:** 2026-06-13
**Scope:** Go server, deterministic sim, store/http ownership transitions, replay.
**Overview:** [`../20260613_v110-overview.md`](../20260613_v110-overview.md)

## Summary

The backend continues to enforce the right boundary: the server owns combat, death, corpse recovery,
market ownership, stash gold, and item mutation. v110 extends an existing store transaction rather
than introducing a parallel upgrade path, which is the right direction.

## Findings

[Strength] Item upgrade authority stays in the store transaction. Cost calculation, gold mutation,
rolled-stat mutation, and max-level rejection are all server-owned.

[Strength] The corpse-recovery work added focused files such as `server/internal/game/corpse.go` and
`server/internal/store/corpse.go`, which is healthier than putting all death recovery logic in
`sim.go`.

[Med] `server/internal/store/repos.go` is now 2723 lines. Upcoming market purchase/delivery work
will likely touch the same area; it should extract helper functions around listing purchase,
stash-gold transfer, and delivery rather than adding one long transaction body.

[Med] `server/internal/game/sim.go` remains 7645 lines after v109. Elite aura work should prefer a
focused aura/effects file and call sites from combat resolution rather than embedding aura logic
directly in broad tick paths.

[Low] `server/internal/http/auth_session_test.go` is 1245 lines. New market HTTP tests should use
local helper functions and consider a focused market route test file if growth is more than a few
cases.

## Recommendations

1. v111 market purchase should be store-first, with atomic buyer/seller gold and item movement.
2. Add market purchase tests in focused helpers and avoid expanding unrelated auth/session cases.
3. Elite aura should reuse existing pack leader metadata and deterministic distance checks.
4. Keep upgrade cost tests rule-derived where the HTTP server has loaded rules available.
