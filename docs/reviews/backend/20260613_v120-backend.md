# v120 Backend Review

Date: 2026-06-13
Scope: Go server, `server/internal/game`, store/economy-adjacent behavior from v111-v120

## Summary

The backend maintained the right ownership model through a feature-heavy batch. Market purchases,
blacksmith upgrades, elite auras, and unique drops are all authored by server state and deterministic
rules. The main concern is not correctness drift; it is continued concentration in the simulator,
rules loader, and broad integration test file.

## Strengths

- Store-backed market purchase and delivery keeps item ownership server-side and avoids client-side
  delivery shortcuts.
- Blacksmith upgrade behavior remains routed through the existing authoritative upgrade endpoint,
  with store handling updated for both legacy flat rolled stats and current nested rolled-stat
  payloads.
- Elite command aura behavior is server-owned and client-readable; radius preview remains display
  only.
- Unique effect reachability is now validated through enabled metadata and compatible templates, so
  enabling the full catalog has a concrete backend proof.

## Findings

### Medium: `sim.go` is still the gameplay accumulation point

`server/internal/game/sim.go` remains about 7.7k lines and still owns unrelated systems such as loot
drop spawning and monster movement/combat loop orchestration. That makes future combat and loot
changes harder to review because unrelated invariants are nearby.

Recommendation: when touching loot generation, extract drop spawning/selection helpers into a
dedicated file. When touching monster behavior, prefer small movement/combat helper files over
adding more loop logic to `sim.go`.

### Medium: `game_test.go` remains a broad integration bucket

`server/internal/game/game_test.go` is about 8.8k lines. v119 added appropriate unique reachability
coverage there because the existing helper surface already lived there, but this file should not be
the default home for every new domain proof.

Recommendation: new economy, unique-effect, and upgrade tests should start in focused files unless
they truly need the whole-session integration harness.

### Low: shared-rule loading and validation are expanding together

`server/internal/game/rules.go` is over 3k lines and continues to grow as the shared catalog surface
expands. That is still manageable, but new itemization/economy catalogs should keep parse,
normalization, and lookup logic grouped by domain.

Recommendation: use focused loader helpers for future unique, economy, and progression rules rather
than extending one long loading path.

## Verification Reviewed

- `make ci` passed for the v120 closeout.
- No separate `go vet` run was performed as part of this review.
