## Summary

The backend stayed authoritative through the inventory-origin market/blacksmith work, Ranger skills,
and the market expiration/withdrawal/audit batch. The most important recent correction is that
accepted market offers now transfer the listed item to the bidder and keep offered items out of the
bidder stash, with store and HTTP coverage documenting the ownership boundary.

## 1. Architecture

[Strength] Market ownership transitions remain server-side. HTTP handlers route publish, purchase,
offer, accept, and cancel actions into store methods rather than trusting client item movement
(`server/internal/http/market.go:110`, `server/internal/http/market.go:184`,
`server/internal/http/market.go:214`).

[Strength] Audit records are written inside the same store transactions as market status and stash
changes. The current store path inserts audit rows during listing creation, cancellation, offer
creation/cancel/accept, expiration refund, and purchase (`server/internal/store/repos.go:1778`,
`server/internal/store/repos.go:1834`, `server/internal/store/repos.go:1896`,
`server/internal/store/repos.go:1942`, `server/internal/store/repos.go:2031`,
`server/internal/store/market_purchase.go:72`).

[Med] Market persistence now spans `repos.go`, `market_purchase.go`, HTTP response shaping, and
migrations. The model is correct enough to build on, but new market features should extract a
market-store file boundary before more transaction branches land in `repos.go` around
`server/internal/store/repos.go:1714`.

## 2. Technical

[Strength] The recent market regression coverage is concrete. Store tests assert accepted offers
move offered items to the seller, deliver the listed item to the bidder, and append audit records
(`server/internal/store/store_test.go:1299`, `server/internal/store/store_test.go:1335`,
`server/internal/store/store_test.go:1344`).

[Med] Listing expiration depends on read-side cleanup through `ListActiveMarketListings`; this is a
reasonable first pass, but future UI or API surfaces that do not call that read path may observe
stale active rows until another listing read occurs (`server/internal/store/repos.go:1714`,
`server/internal/store/repos.go:2053`). A small follow-up should make expiration explicit from
summary/detail paths or document the read-side cleanup contract.

[Low] The deterministic sim is still protected by process and linting, and no new `time.Now` or
`math/rand` hazards were found under `server/internal/game`. The only `panic` hits are existing
sim setup/active-level invariants (`server/internal/game/sim.go:389`,
`server/internal/game/sim.go:1874`).

## 3. Maintainability

[Med] `server/internal/game/game_test.go` remains the largest backend test file at 8995 lines. The
Ranger and unique work has enough domain identity now that new proofs should start in focused files
instead of extending the shared integration bucket.

[Med] `server/internal/game/sim.go` remains a 7745-line coordinator. v121-v130 did not materially
grow it, which is good. The next combat or loot slice should keep adding focused helper files rather
than re-opening broad sim sections.

[Low] The store package is becoming the current growth area. Market methods in `repos.go` now sit
beside unrelated character/session/stash logic, so a `market_repo.go` extraction is a good
maintenance slice before advanced market restrictions or unique-market rules.

## 4. Documentation

[Strength] v128-v130 as-built docs record exactly what changed in market expiration, withdrawal,
and audit rows, and the proof commands are focused on store/http/replay.

[Low] The v128-v130 docs prove backend behavior, but no player-facing audit browser exists yet.
That is fine for this batch; it should stay a non-goal until an explicit inspection UI slice.

## Top 5 backend refactors

1. Extract market store methods from `repos.go` into a focused market repository file.
2. Add an explicit market expiration service/helper callable by summary/detail paths, not only active-list reads.
3. Start new Ranger and unique-effect tests in focused backend test files.
4. Keep purchase/offer/listing response DTO conversion in HTTP, but avoid growing `market.go` with audit browsing until the route is specced.
5. Preserve audit writes in the same transaction as ownership changes for every future market action.
