## Summary

The backend closed several v130 risks while preserving the server-authoritative boundary. Unique
effects and named unique packages now have focused server files, the deterministic town unique chest
proves obtainability, and v139 removed stale market expiration reads from summary and seller-offer
paths. The main backend concern is now not correctness of these shipped flows, but where the next
work lands: `repos.go`, `sim.go`, and `game_test.go` remain large enough that new domains should not
default there.

## 1. Architecture

[Strength] Market expiration read freshness is now explicit. Active listings, seller offer reads,
and summary reads all trigger `ExpireMarketListings` before returning active market state
(`server/internal/store/repos.go:1710`, `server/internal/store/repos.go:1957`,
`server/internal/store/repos.go:2143`).

[Strength] Market ownership transitions and audit writes remain in store transactions. Listing
publish deletes the seller stash item and writes `listing_published`; offer submit reserves bidder
items and writes `offer_submitted`; acceptance delivers both sides and writes `offer_accepted`;
expiration restores the listing item, refunds active offers, and writes `listing_expired`
(`server/internal/store/repos.go:1771`, `server/internal/store/repos.go:1778`,
`server/internal/store/repos.go:1891`, `server/internal/store/repos.go:1896`,
`server/internal/store/repos.go:2004`, `server/internal/store/repos.go:2034`,
`server/internal/store/repos.go:2078`, `server/internal/store/repos.go:2101`).

[Med] Market persistence is still spread across a broad store coordinator. The market block in
`repos.go` spans listing, cancel, offer, accept, expiration, audit, and summary behavior
(`server/internal/store/repos.go:1710` through `server/internal/store/repos.go:2164`) while purchase
logic already lives separately in `market_purchase.go`. The next economy slice should extract this
domain before adding richer restrictions or audit APIs.

## 2. Technical

[Strength] v139 added direct regression tests for the previously noted stale-read contract. The
tests force a listing into the past, call summary or seller-offer reads, and assert seller item
restoration, bidder item refund, zero active counts/offers, and `listing_expired` audit
(`server/internal/store/market_expiration_read_test.go:75`,
`server/internal/store/market_expiration_read_test.go:89`,
`server/internal/store/market_expiration_read_test.go:103`).

[Strength] Unique item behavior is now more cohesive than at v130. Unique effect logic lives in
focused files such as `unique_effects.go`, `unique_resource_effects.go`,
`unique_survival_effects.go`, and `unique_chest.go`, rather than being only another `sim.go`
extension (see `docs/CODEMAP.md:10`).

[Low] Determinism risk remains well controlled by process. `make ci` passed after v139, including
determinism lint, Go tests, protocol bot + replay, Godot client bot scenarios, and client smoke.

## 3. Maintainability

[Med] `game_test.go` and `sim.go` are still the top backend hotspots at 9116 and 7801 lines. Their
current size is grandfathered by the ratchet, but future gameplay slices should start in focused
domain files and tests unless a plan explicitly explains why the coordinator must change.

[Med] `repos.go` grew slightly in v139 to close a correctness gap, and the plan documented that
exception. The ratchet allowed it, but the design debt remains: split market store methods before
adding new market capabilities.

[Strength] `make maintainability` is now a CI gate and prints the grandfathered trend. At this
baseline it passes with 33 grandfathered files and 68778 lines.

## 4. Documentation

[Strength] `docs/CODEMAP.md` gives future agents a domain-to-files loading map, including market,
unique items/effects, stash, combat, realtime, replay, and tooling rows (`docs/CODEMAP.md:7`).

[Low] The market expiration behavior is now covered by specs/as-built and tests, but no user/admin
audit browser exists. That remains correctly deferred until an explicit inspection UI slice.

## Top 5 backend refactors

1. Extract market listing/offer/expiration/audit methods from `repos.go` into a focused market
   repository file.
2. Keep new unique/economy tests in focused files, following `market_expiration_read_test.go`.
3. Split the next combat or dungeon behavior out of `sim.go` before adding more coordinator logic.
4. Continue shrinking `game_test.go` by moving domain tests into focused files as touched.
5. Preserve market audit writes inside the same transaction as every future ownership mutation.
