## Summary

The backend stayed server-authoritative through the v151-v159 feature batch. v159 made a small but useful extraction by moving interactable activation out of `sim.go`, and the elite-objective chest rule now rejects early activation before range and chest mutation paths run. The backend risks are familiar and still current: `game_test.go` remains the largest repo file, `populateDungeonLevel` is a cohesive dungeon-generation extraction candidate inside `sim.go`, and `rules.go` still carries broad loader/validator responsibility.

## 1. Architecture

[Strength] Interactable activation now has a focused server-side home. `activateInteractable` routes town services, test unique chests, already-open chest checks, elite objective gates, and treasure chest loot from `server/internal/game/interactables.go:3`, instead of adding another branch to the main sim coordinator.

[Strength] The v159 elite-objective gate preserves authority and fails early. `handleAction` rejects locked objective chests with `elite_objective_incomplete` before distance checks and activation side effects (`server/internal/game/handlers.go:144`, `server/internal/game/handlers.go:154`). The runtime objective identity lives in `LevelState` as `eliteObjectiveChestIDs` (`server/internal/game/level.go:5`, `server/internal/game/level.go:15`).

[Med] Dungeon level population is the clearest next `sim.go` extraction. `populateDungeonLevel` still creates stairs, teleporters, chests, objective chest runtime IDs, loose loot, generated monsters, and position fallback behavior in one function (`server/internal/game/sim.go:1238`, `server/internal/game/sim.go:1274`). That is a cohesive enough domain to move without a speculative abstraction.

## 2. Technical

[Strength] Objective completion is deterministic and data-backed. `eliteObjectiveChestLocked` sorts entity IDs before scanning generated pack leaders and unlocks when any leader on that level has `hp <= 0` (`server/internal/game/interactables.go:57`, `server/internal/game/interactables.go:62`). That matches the documented v159 scope and avoids map-iteration nondeterminism.

[Strength] The implementation kept chest behavior on the existing path. Once unlocked, objective chests use the same `opened`, `OpenedChests`, entity update, event, and loot spawning path as ordinary treasure chests (`server/internal/game/interactables.go:44`).

[Med] Equipment/stat preview helpers still live in `sim.go`. `equipPreviewForItemWithSlot` and nearby stat calculation helpers keep non-tick logic in the broad sim file (`server/internal/game/sim.go:6072`). This is not urgent, but any future paper-doll or requirement slice should avoid growing that area.

## 3. Maintainability

[Strength] v159 lowered `sim.go` from 7045 to 7022 lines and created `interactables.go`. The extraction is small, but it follows the desired pattern: move a real behavior domain, keep tests green, and lower the ratchet baseline in the same slice.

[High] `game_test.go` remains the largest backend risk at 9129 lines. The generic treasure chest test still lives there and now needs an elite-objective fixture toggle (`server/internal/game/game_test.go:5488`), while the new objective behavior correctly landed in a focused dungeon objective test. Future touched tests should follow the focused-file pattern.

[Med] `rules.go` is still broad at 3268 lines. The top-level `Rules` struct spans many catalog families (`server/internal/game/rules.go:13`), and skill validation/helper logic reaches the end of the file (`server/internal/game/rules.go:3000`). New rule domains should prefer catalog-specific validators when they become non-trivial.

## 4. Documentation

[Strength] The v159 spec, plan, and as-built record the intentional non-goal that opening every elite leader is deferred. `PROGRESS.md` also carries the deferred item under Combat so the simplified completion rule is not mistaken for final encounter design.

[Low] CODEMAP did not require a broad rewrite for v159 because the new file stays inside the existing `server/internal/game` domain, but future larger extractions should update it when routing changes.

## Top 5 backend refactors

1. Extract dungeon runtime population from `sim.go`, centered on `populateDungeonLevel` and objective chest runtime ID ownership.
2. Move touched `game_test.go` regions into focused domain test files, starting with chest/interactable and dungeon objective coverage.
3. Split the next catalog-specific validator out of `rules.go` when a shared rule family grows.
4. Extract equipment/stat preview helpers from `sim.go` only when an equipment or paper-doll slice already touches them.
5. Keep server-side rejection paths covered by both focused Go tests and one protocol bot proof when they affect player-visible intent errors.
