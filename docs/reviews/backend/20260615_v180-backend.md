# Backend review v180

Date: 2026-06-15
Baseline: v180 — upgrade resource drop

## What improved

- Boss pattern validation is now isolated in `server/internal/game/boss_pattern_rules.go`, with rule validation covering required phases, timing, telegraph shape, summon monster IDs, summon count, and active-hit consistency.
- Mana regeneration is explicit in the simulation event stream via `player_mana_regenerated` in `server/internal/game/sim.go:1082`, and the protocol bot scenario `tools/bot/scenarios/71_mana_regeneration.json:20` proves the event and value.
- Upgrade shards are now real catalog drops. The shared loot table includes `upgrade_shard` in `shared/rules/loot_tables.v0.json:15`, and `tools/bot/scenarios/72_upgrade_resource_drop.json:12` proves pickup through the protocol bot.
- Interactable tests were separated into `server/internal/game/interactables_test.go:33` and `server/internal/game/interactables_test.go:70`, reducing pressure on the broad game test file.

## Risks

1. `server/internal/game/game_test.go` remains a broad integration-test bucket at 8,814 lines. The newest refactor moved door and chest proofs out, but boss-floor, dungeon, ranged, and loot behavior still share the same test file. Future domain work should create focused test files instead of appending here.

2. `server/internal/game/sim.go` remains a high-risk coordinator at 6,642 lines. v171-v180 additions mostly respected data ownership, but any new combat/resource/status behavior can still spread across tick processing, event emission, entity mutation, and test helpers.

3. Store persistence remains concentrated in `server/internal/store/repos.go` at 2,315 lines. The market extraction helped, but future account-stash, upgrade, or listing changes should continue moving cohesive store operations into domain files.

4. `server/internal/game/rules.go` is still 3,222 lines. Boss-pattern validation moved out, but remaining catalog validation can continue to split by domain when those domains are touched.

## Recommended backend work

- Move the next test cluster out of `game_test.go` when a slice touches that behavior. Good candidates are boss-floor gates, ranged loot/action reach, and dungeon transition cases.
- Keep new upgrade-resource economy work data-driven in shared rules. Do not hardcode drop rates, costs, or item IDs in simulation logic beyond catalog identifiers and contract plumbing.
- Split store operations opportunistically when adding upgrade-resource spending or stash mutations.
