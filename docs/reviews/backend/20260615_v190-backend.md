## Summary

The v181-v190 backend work kept the server-authoritative boundary intact while adding set items, companions, elite minion pack behavior, rarity/affix expansion, and Paladin defensive fixes. Companion actors are protocol-visible as `companion` entities in v8 snapshots (`shared/protocol/session_snapshot.v8.schema.json:80`) and bot-covered by the v182-v185 scenarios (`tools/bot/scenarios/73_companion_ai_foundation.json:8`, `tools/bot/scenarios/76_companion_rank_scaling_and_limits.json:20`).

The main backend risks are now narrower and more actionable than the v180 review: companion/elite-minion tuning has small code-owned constants, monster-rarity validation still hardcodes values that otherwise live in rules, and large coordinators remain the obvious places where future slices can accidentally add more coupling.

## 1. Architecture

- **[Strength] Authoritative state remains server-owned.** Companion spawning, revive conversion, movement, targeting, combat, and kill resolution all live in `server/internal/game/companion_ai.go`, with companion damage emitted through the same combat event path as monsters (`server/internal/game/companion_ai.go:427`).
- **[Strength] Deterministic ordering is preserved in new AI loops.** Companion and elite-minion target scans use sorted entity IDs (`server/internal/game/companion_ai.go:232`, `server/internal/game/companion_ai.go:264`, `server/internal/game/elite_minion_ai.go:10`).
- **[Med] Companion/elite-minion AI tuning is code-owned.** `companionAssistRadius`, `companionFollowDistance`, and `companionFollowStopRadius` are constants in Go (`server/internal/game/companion_ai.go:5`) and elite minions reuse the same follow radii (`server/internal/game/elite_minion_ai.go:45`, `server/internal/game/elite_minion_ai.go:61`). These values affect gameplay feel and should move to shared rules before more companion types or minion roles are tuned.

## 2. Technical

- **[Strength] The v190 defensive stat fix is properly exposed through the authoritative derived-stat view.** `DerivedStatsView` now includes block and evade fields (`server/internal/game/derived_stats.go:8`, `server/internal/game/derived_stats.go:14`), and `DerivedStatsView()` returns effective combat stats rather than only base character formulas (`server/internal/game/derived_stats.go:23`).
- **[Strength] Rare and skill affixes have direct simulation tests.** Rare combat rolls are tested against derived stats and combat resolution (`server/internal/game/rare_combat_affixes_test.go:8`, `server/internal/game/rare_combat_affixes_test.go:40`), while skill-affix tests verify mana-cost and cooldown reduction (`server/internal/game/skill_affixes_test.go:5`).
- **[Med] Some rule validation still pins tuning in Go code.** `validateMonsterRarities` hardcodes expected rarity IDs, weights, offsets, visual scales, and colors (`server/internal/game/rules.go:2247`). That is defensible as a golden-like gate, but it is currently mixed into the runtime loader rather than a named golden/schema validation path, so ordinary tuning changes can require code edits.

## 3. Maintainability

- **[Strength] The pre-review paydown reduced and locked a large-test baseline.** `game_test.go` is now 7,888 lines and `.maintainability/file-size-baseline.tsv` was lowered accordingly in `3f5f8a47`.
- **[Med] The largest backend files are still large enough to distort future slice plans.** `game_test.go`, `sim.go`, and `rules.go` remain the top three Go pressure points. New companion, affix, or itemization slices should create focused files instead of adding to these broad coordinators.
- **[Low] Set-item validation is well isolated but could use Python-side parity.** Go validates catalog semantics in `server/internal/game/set_items.go:35`, while `shared/rules/set_items.v0.schema.json:15` covers the schema shape. A standalone `tools/validate_set_items.py` would keep `validate_shared.py` from absorbing the next set-item cross-check.

## 4. Documentation

- **[Strength] v181-v190 have specs/plans/as-built entries recorded in `PROGRESS.md`.**
- **[Low] ADR-0010 is still a future mercenary ADR, while current companions are simpler server-owned pets/revives.** That distinction is clear enough today, but the next companion slice should update ADR/design language if player-derived mercenaries stop being "future direction."

## Top 5 backend refactors

1. Move companion follow/assist radii and target-acquisition tuning into shared rules, with Go tests deriving expectations from those rules.
2. Convert `validateMonsterRarities` hardcoded expected values into an explicit golden/schema-owned validation path.
3. Extract the next touched cluster from `server/internal/game/game_test.go`; companion, rarity/affix, and dungeon/boss clusters are good candidates.
4. Keep new item/set/affix semantics in focused files, not `sim.go` or `rules.go`, unless the change is loader plumbing.
5. Split the next store mutation domain out of `server/internal/store/repos.go` when item upgrades or companion persistence becomes durable.
