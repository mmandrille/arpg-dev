## Summary

The backend remains healthy at v250. The selected `$autoloop` queue shipped several client-facing
views, but the backend also carried real authoritative changes: market listing expiration exposure,
blacksmith recipe selection and validation, companion combat stats in entity views, and ranged
monster projectiles that can target and damage engaged companions. The full batch `make ci` passed
after v250, and review-time `make validate-shared` plus `make maintainability` passed.

The most important prior backend follow-up changed shape. Companion follow/assist tuning has moved
into `main_config`; the remaining issue is semantic coupling because elite minions reuse companion
tuning names. Monster-rarity validation no longer appears to pin exact expected weights/colors in
Go, so the old v232 recommendation should be closed rather than carried forward.

## 1. Architecture

- **[Strength] Server authority held through market and blacksmith changes.** Market active reads
  filter by server-side listing status and `expires_at > now()` (`server/internal/store/market_repo.go:16`),
  and HTTP responses serialize the authoritative expiration timestamp (`server/internal/http/market.go:35`,
  `server/internal/http/market.go:391`). Blacksmith upgrade requests carry `recipe_id` through the
  server route contract (`server/internal/http/account_stash.go:73`) and server tests cover the
  `weapon_honing` success, non-weapon conflict, and unknown-recipe rejection paths
  (`server/internal/http/blacksmith_recipe_test.go:38`).
- **[Strength] Companion combat stats remain server-owned.** Entity views expose optional
  `combat_stats` (`server/internal/game/types.go:58`) and populate companion views from sim state
  (`server/internal/game/sim.go:5548`), which keeps the mercenary panel display downstream of the
  authoritative sim.
- **[Strength] Ranged companion damage stayed in authoritative combat paths.** Monster attacks can
  resolve against companion targets (`server/internal/game/sim.go:3281`), projectile collision uses
  `damageCompanionByMonster` (`server/internal/game/monster_companion_combat.go:99`), and the focused
  test asserts archer-owned projectiles target and damage companions (`server/internal/game/ranged_companion_projectile_test.go:31`).
- **[Med] Elite-minion movement still reuses companion tuning names.** The values are now
  data-driven through `main_config` (`server/internal/game/rules.go:61`), but elite minions use
  `CompanionFollowStop` and `CompanionFollowDistance` for leader formation behavior
  (`server/internal/game/elite_minion_ai.go:45`, `server/internal/game/elite_minion_ai.go:61`).
  That is acceptable short-term reuse, but it couples two AI domains to one tuning vocabulary.

## 2. Technical

- **[Strength] Validation and CI are green.** The selected v241-v250 batch `make ci` passed after
  v250 in 10m27s. Review-time `make validate-shared` passed 1,295 checks plus `tools/validate_codemap.py`,
  and `make maintainability` passed the file-size, extraction-coupling, and progress-dashboard gates.
- **[Strength] Prior monster-rarity validation debt is resolved or no longer supported by current
  evidence.** `validateMonsterRarities` now checks shape, uniqueness, ranges, colors, and positivity
  generically (`server/internal/game/rules.go:2319`); it does not hardcode a specific rarity set,
  weight table, or color catalog.
- **[Low] A few batch-specific backend proofs still rely on broader integration checks.** v247 and
  v248 added focused Go tests for mercenary stats and ranged companion projectile behavior, but the
  end-to-end client bot proofs are still the stronger signal for the combined server/client flows.

## 3. Maintainability

- **[Strength] Backend file-size debt stayed inside the ratchet.** `make maintainability` reports
  34 grandfathered files / 63,983 lines, and current large backend files remain grandfathered in
  `.maintainability/file-size-baseline.tsv`.
- **[Med] Large backend coordinators remain expensive.** `game_test.go`, `sim.go`, `rules.go`,
  `repos.go`, and `store_test.go` are still well above the 600-line target. Future backend feature
  work should keep extracting or bypassing these files rather than adding new domains to them.
- **[Low] Main-config promotion improved data ownership but left naming pressure.** Companion
  follow/assist values are validated as positive config (`server/internal/game/rules.go:1007`), and
  tests prove config drives companion spawn/travel (`server/internal/game/companion_ai_test.go:84`).
  Separate elite-minion names can be planned when AI tuning is next touched.

## 4. Documentation

- **[Strength] Backend-relevant ADRs still match implementation direction.** ADR-0001 keeps combat,
  loot, inventory, progression, and persistence server-owned; ADR-0012 requires upgrade results to
  remain server-authored and deterministic.
- **[Low] Batch proof docs lag the actual CI result.** The v241-v250 lifecycle/as-built entries still
  say batch `make ci` is pending, even though the batch passed after v250. That is a process-doc
  fix, not a backend correctness issue.

## Top 5 backend refactors

1. Close the v241-v250 lifecycle/as-built batch-CI wording so docs match the green CI baseline.
2. Split the next touched assertion cluster out of `server/internal/game/game_test.go`.
3. Split or narrow the next touched persistence cluster in `server/internal/store/repos.go`.
4. Plan separate elite-minion follow tuning names in shared rules when AI tuning is next in scope.
5. Keep ranged companion targeting covered by focused Go tests before broader bot/CI runs.
