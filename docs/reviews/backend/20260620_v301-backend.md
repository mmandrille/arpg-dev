## Summary

The v295-v301 feature batch was intentionally client-side and did not move gameplay authority out of
the Go server. Input buffering, attack move, local windup/recovery presentation, hit stop, movement
visual smoothing, command retarget grace, and melee lunge all reuse existing `action_intent` /
`move_to_intent` dispatch while the server remains the source of truth for position, reach, damage,
death, loot, and replay.

Backend verification is green at the component level: shared validation, asset validation,
determinism lint, Go tests, Go vet, Python tools, protocol bot plus replay, and the full client bot
suite all passed during the post-loop proof or this review. Raw `make ci` could not complete step
8 in this isolated worktree because `make db-up` tried to own a Postgres container already running
from the main worktree.

## 1. Architecture

- **[Strength] Server authority held through client-feel work.** Local attack presentation starts
  immediately, but the client still sends the existing attack intent and starts local recovery only
  after the same dispatch path (`client/scripts/main.gd:2766`, `client/scripts/main.gd:2767`,
  `client/scripts/main.gd:2768`). Attack move computes a client approach point for input feel, then
  sends the existing `move_to_intent` instead of changing server pathing (`client/scripts/main.gd:2788`,
  `client/scripts/main.gd:2791`).
- **[Strength] Replay/determinism boundaries remain enforced.** `make lint-determinism` passed, and
  the CI script still gates deterministic game code before server-dependent bot/replay checks
  (`scripts/ci.sh:263`, `scripts/ci.sh:267`, `scripts/ci.sh:337`, `scripts/ci.sh:361`).
- **[Strength] Backend contract churn stayed near zero for this batch.** The Movement / Combat
  Fluidity changes were absorbed by client helpers and client bot scenarios; no new public protocol,
  persistence, or server simulation concepts were required.

## 2. Technical

- **[Strength] Current backend checks are green.** `cd server && go test ./...` passed, including
  store tests, and `cd server && go vet ./...` exited cleanly. `make validate-shared`,
  `make validate-assets`, and `make lint-determinism` also passed during this review.
- **[Strength] The v284 broad-suite flakes are no longer current release blockers.** The post-loop
  `SCENARIO=all` client bot sweep reported 85 passed / 0 failed, and `mercenary_recovery_ui` passed
  again after the stale companion cleanup fix. The old `door_fog_toggle` flake did not reappear.
- **[Med] Raw CI is not worktree-isolated enough.** `scripts/ci.sh` unconditionally runs `make db-up`
  before server-dependent steps (`scripts/ci.sh:274`, `scripts/ci.sh:278`). In this run, an existing
  healthy `arpg-postgres` container from the main worktree caused step 8 to fail before the server
  could start. The codebase still needs a raw `make ci` path that works from isolated worktrees.

## 3. Maintainability

- **[Strength] The ratchets are still useful guardrails.** `make maintainability` passed with 34

- **[Med] Backend size debt remains concentrated.** `game_test.go`, `sim.go`, `rules.go`,

## Top 5 backend refactors

1. Make `make ci` server bootstrap worktree-safe: detect/reuse an existing healthy local DB or use
   per-worktree Compose/container identity so raw CI can run from isolated branches.
2. Split the next touched backend assertion cluster out of `server/internal/game/game_test.go`.
3. Split the next touched dungeon-generation subdomain out of `server/internal/game/dungeon_gen.go`.
4. Keep future movement/combat authority changes server-owned and replay-backed; do not promote the
   v295-v301 client feel helpers into gameplay truth.
5. Keep Go tests deriving expectations from shared rules when future balance or movement tuning
   changes touch combat formulas.
