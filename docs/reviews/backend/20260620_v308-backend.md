## Summary

The v302–v308 batch is the strongest data-driven slice batch in the repo to date. Flying navigation,
barbarian leap obstacle-crossing, water/hole hazards, and line-of-sight blocking are all genuinely
server-owned outcomes — the client renders resulting state only. New mechanics read every
balance-sensitive value from `shared/rules/` (worlds, dungeon_generation, monsters, skills) with
essentially no hardcoded tuning. The obstacles golden is a real deterministic contract, new tests
derive expectations from loaded rules, and `go test ./...`, `go vet ./...`, `make lint-determinism`,
and `make maintainability` all pass on the current commit. The defects are structural/cosmetic, not
correctness: a dead `switch` in `obstacle_blocking.go`, ~90% duplication between the water and hole
placers, one gameplay-feel decision baked into Go instead of data, and a stale `docs/CODEMAP.md` that
omits every new backend file. Both v301 backend recommendations are confirmed (worktree-safe CI
RESOLVED; the monolith-split rec still effectively OPEN for `dungeon_gen.go`).

## 1. Architecture

- **[Strength] Authority boundary held across all four features.** Fog/LoS is server-computed:
  `blocksFogLineOfSight` walks walls and hides monsters from the delta
  (`server/internal/game/fog_of_war.go:200`, `server/internal/game/fog_of_war.go:214`); the client
  only consumes a `fog_of_war` block and renders an overlay. Flying collision is resolved server-side
  via `monsterNavigationDef` + `monsterObstacleBlocksMovement`, and *both* pathfinding and live
  movement route through the same predicate (`server/internal/game/sim.go:3427`,
  `server/internal/game/sim.go:3412`) — no client/server divergence. `BlocksLineOfSight` is emitted as
  render metadata on `WallView` (`server/internal/game/sim.go:6268`), not client-evaluated.
- **[Strength] Intent-dispatch invariant honored.** Leap/charge ride the existing
  `cast_skill_intent` / `channel_skill_intent` handlers in the `inputHandlers` registry
  (`server/internal/game/handlers.go:44`), dispatched to `handleMobilitySkillCast`
  (`server/internal/game/handlers.go:1219`). No new intent type and no new `case` in `applyInput`.
- **[Strength] New dungeon files are highly cohesive,** each a small single-purpose module
  (`dungeon_water.go`, `dungeon_holes.go`, `dungeon_obstacle_variety.go`,
  `monster_navigation_traits.go`, `obstacle_blocking.go`), unit-testable in isolation.
- **[Med] `dungeon_water.go` and `dungeon_holes.go` are ~90% duplicated.** `placeDungeonWater` /
  `placeDungeonHoles`, `randomWaterObstacles` / `randomHoleObstacles`, and their validators differ
  only by the rule field selected and the `obstacleKind` constant
  (`server/internal/game/dungeon_water.go:40`, `server/internal/game/dungeon_holes.go:40`). Two copies
  of one algorithm; a single parametric `placeFloorFeature(kind, sizeRules, …)` collapses both. Tiny
  files today, but copy-paste rot if a third hazard lands.
- **[Low] Companion collision is trait-blind by silent omission.** `companionPositionBlocked` calls
  the trait-blind `obstacleBlocksMovement` (`server/internal/game/companion_ai.go:398`), so a
  companion can never fly. Defensible design, but it is an implicit assumption rather than an explicit
  `grounded` config.

## 2. Technical correctness

- **[Strength] Determinism holds.** Hazard/obstacle placement uses seeded RNG with level + attempt
  folded into the seed string (`server/internal/game/dungeon_water.go:47`,
  `server/internal/game/dungeon_holes.go:47`); `make lint-determinism` passes on `./internal/game/...`.
  No `time.Now()`, no `math/rand`. New hot-path map ranges use `sortedEntityIDs`
  (`server/internal/game/mobility_skills.go:62`, `server/internal/game/fog_of_war.go:222`).
- **[Strength] Reachability re-validated after placement.** Both placers append candidate walls, re-run
  `validateGeneratedDungeonReachability`, retry up to `MaxAttempts`, and roll back on failure
  (`server/internal/game/dungeon_water.go:54`). No path to an unreachable down-stair.
- **[Strength] Leap landing safety.** The sweep only advances past obstacles it is allowed to ignore
  and stops at hard walls (`server/internal/game/mobility_skills.go:37`); the cast is rejected when the
  endpoint equals the start (`server/internal/game/mobility_skills.go:97`).
- **[High] Dead `switch` in `obstacle_blocking.go:19`.** `obstacleBlocksMovement` has
  `case obstacleKindWater, obstacleKindHole: return true` and `default: return true` — both branches
  return the same value, so the `switch` is noise that *implies* water/hole are special-cased here when
  they are not. Reduce to `return true` with a comment that hazard-vs-trait differentiation lives in
  `monsterObstacleBlocksMovement` / `skillMobilityIgnoresObstacleKind`. Not a runtime bug; a
  correctness *smell* that will mislead the next reader. **(Verified by direct read.)**
- **[Low] Fractional same-min/max size config can escape its band.** `randomWaterObstacle` uses
  `randomIntRange(rng, Ceil(MinSize.X), Floor(MaxSize.X))` (`server/internal/game/dungeon_water.go:104`);
  a future `min=max=2.5` config would `Ceil→3, Floor→2` and silently return `min`. The guard prevents a
  panic but does not honor the rule. Only bites on fractional configs.

## 3. Data-driven config compliance

- **[Strength] Water/hole geometry, leap, flying, and solid-kind weights are all data.** Water/hole
  `enabled`/`max_attempts`/`target_count`/`min_size`/`max_size` come from
  `shared/rules/dungeon_generation.v0.json` (`server/internal/game/dungeon_water.go:9`); leap
  `range_base` / `ignore_obstacle_kinds` / `impact_radius` / `stun_duration_ticks` from
  `shared/rules/skills.v0.json` (`server/internal/game/mobility_skills.go:5`); flying
  `navigation_trait` from `shared/rules/monsters.v0.json:301` validated by `validMonsterNavigationTrait`
  (`server/internal/game/rules.go:1559`); solid-kind weights from `dungeon_generation.v0.json` via
  `chooseSolidObstacleKind` (`server/internal/game/dungeon_gen.go:279`).
- **[Med] One gameplay-feel decision is baked into Go.** *Which* solid kinds block line of sight
  (`rock`/`column` block, `rubble` does not) is hardcoded in a Go `switch`
  (`server/internal/game/dungeon_obstacle_variety.go:53`). Per the Data-Driven Configuration Policy this
  should be a per-kind `blocks_line_of_sight` flag in `dungeon_generation.v0.json`, or carry an explicit
  code-ownership note in the spec. The sweep granularity `0.25`
  (`server/internal/game/mobility_skills.go:32`) and retry budget `32` are defensible implementation
  details, but should be acknowledged.

## 4. Test quality

- **[Strength] The obstacles golden is a real deterministic contract.**
  `dungeon_obstacles_golden_test.go` pins exact wall IDs/positions/kinds (legitimate replay
  determinism) *and* asserts semantic minimums (`MinimumWaterCount`, `MinimumHoleCount`), shape/kind
  set coverage, and full reachability (`server/internal/game/dungeon_obstacles_golden_test.go:97`). The
  `-update` path normalizes nil slices to `[]T{}`.
- **[Strength] New tests derive from loaded rules, not copied constants.**
  `dungeon_obstacle_variety_test.go:17` mutates `loadRules(t).DungeonGeneration` and proves generation
  follows the override; leap tests read `sim.rules.Skills["leap"]`
  (`server/internal/game/mobility_obstacle_crossing_test.go:7`); the nav-trait test proves the flying
  path is shorter and the grounded path detours (`server/internal/game/monster_navigation_traits_test.go:5`).
  Negative/parity coverage is present (dash still hard-blocks on water; flying still hits normal walls).
- **[Low] Leap landing thresholds are lab-geometry magic numbers** (`landed.X <= 5.6` /
  `>= 3.25`, `server/internal/game/mobility_obstacle_crossing_test.go:16`) derived from the
  `barbarian_leap_obstacle_lab` preset — acceptable, but a comment tying them to the preset would slow
  rot.
- **[Low] No end-to-end fog-hide-by-kind delta test.** `blocksFogLineOfSight` is unit-covered, but no
  Go test proves a `column`/`rock` hides a monster from the *protocol delta* while `rubble`/`water` does
  not; that contract currently rests on the bot scenario only.

## 5. Maintainability

- **[Strength] Every new file is well under 600 lines** and started in its own domain file rather than
  bloating `dungeon_gen.go` — honors the "new domains in focused files" ratchet rule.
- **[Strength] `game_test.go` was not touched and is slightly below baseline** (7,819 vs 7,827) — the
  v301 "stop growing the monolith test" guidance met by avoidance.
- **[Med] The batch consumed nearly all the drift budget on three coordinators without lowering a
  baseline.** Baseline → current: `rules.go` 3,303 → 3,328 (**+25, exactly at the cap**), `types.go`
  721 → 746 (**+25, at the cap**), `dungeon_gen.go` 1,061 → 1,085 (+24, one line from the cap)
  (`.maintainability/file-size-baseline.tsv:16`, `:19`, `:23`). `make maintainability` passes, but
  `rules.go` and `types.go` now have **zero headroom**: the next slice that touches either must extract
  before it adds.

## 6. Documentation

- **[High] `docs/CODEMAP.md` omits every new backend file.** The "Dungeon generation" row
  (`docs/CODEMAP.md:15`) still lists only `dungeon_gen.go, level.go, pathfind.go` — missing
  `dungeon_water.go`, `dungeon_holes.go`, `dungeon_obstacle_variety.go`, `obstacle_blocking.go`, and
  `monster_navigation_traits.go`, with no row at all for fog/line-of-sight (`fog_of_war.go`) or mobility
  skills (`mobility_skills.go`). `tools/validate_codemap.py` only checks that *listed* paths exist, not
  that all files are listed, so CI did not catch it. An agent loading "dungeon generation" context today
  misses the five files implementing the headline feature batch. **(Verified by direct read.)**
- **[Strength] Per-slice SDD docs are complete** for v302–v308 (spec + plan + as-built each); the gap is
  specifically the cross-cutting index.

## Top 5 backend refactors

1. **Fix the dead `switch` in `obstacle_blocking.go:19`** — reduce `obstacleBlocksMovement` to
   `return true` with a comment pointing at where differentiation actually lives. (Lowest-risk.)
2. **Collapse `dungeon_water.go` + `dungeon_holes.go` into one parametric `dungeon_floor_features.go`** —
   removes ~90 duplicated lines; a third hazard becomes a config change, not a file copy.
3. **Update `docs/CODEMAP.md`** to add the five new files to the Dungeon-generation row and add a
   fog/LoS + mobility row, then strengthen `tools/validate_codemap.py` to flag `internal/game/*.go`
   files present on disk but absent from the index so this drift fails CI.
4. **Move "which solid kinds block LoS" out of the Go `switch`
   (`dungeon_obstacle_variety.go:53`) into `dungeon_generation.v0.json`** as a per-kind flag, or add the
   explicit code-ownership spec note the policy requires.
5. **Pre-emptively extract obstacle/nav rule structs from `rules.go` / `types.go`** into a
   `dungeon_generation_rules.go` with their validators, lowering both baselines off their +25 caps
   before the next slice forces it.
