## Top 5 backend refactors

**1. Fix the failing test suite before the next slice** [High]
Seven tests are failing as of HEAD. The pickup/auto-pickup family (`TestItemPickupFromInventory`, `TestGoldAutoPickupWalkingIntoTownGold`, `TestResourceWalletAutoPickupWalkingIntoShard`, `TestDoorLabClosedDoorPreventsPassageUntilActivated`) and the `TestReconstructFromInputsWithPassiveGoldAutoPickup` replay failure smell like a single causal regression — likely the room-layout or movement changes altered world geometry that these tests depended on implicitly. The collision failures and `TestDungeonObstacleGeneration` need separate investigation. Do not start v332 feature work until `go test ./...` is green; a failing test suite masks future regressions and breaks the replay determinism contract.

**2. Extract type clusters from `rules.go` into domain-focused files** [High]
`/Users/mmandrille/git/arpg-dev/server/internal/game/rules.go` at 3 308 lines contains 80 type definitions spanning at least 6 distinct domains: dungeon generation (lines 151–350), combat (84–132), skill trees (552–720), shop definitions (877–948), monsters (738–819), and loot tables (820–876). None of these type clusters imports the other. A mechanical split: create `rules_dungeon_types.go`, `rules_skill_types.go`, `rules_shop_types.go`, `rules_monster_types.go`, leaving `rules.go` for the `Rules` struct, `Load()`, and cross-cutting config. This is the highest-leverage maintainability move available without touching logic.

**3. Extract from `sim.go` using a typed tick context** [Med]
`/Users/mmandrille/git/arpg-dev/server/internal/game/sim.go` at 6 541 lines is still 11× the target. The movement extraction pattern from v332 (`movement.go`) proved the approach works. Next candidates: the `Tick()` orchestration loop itself and the fog-visibility integration added in v329–v331 (currently in `sim.go`). A typed `TickContext` or `SimContext` struct would let tick-phase helpers be extracted to focused files without the `helpers=globals()` anti-pattern.

**4. Document the movement-speed formula as an ADR** [Med]
The formula `classBase × DEX_multiplier × (1 + gear_percent/100)` is a multi-factor multiplicative pipeline that now affects replay determinism, bot scenario budgets (bot timing constraints depend on move speed), and test assertions. There is no ADR for it. Write a short ADR (`docs/adr/ADR-00NN-movement-speed-formula.md`) covering: the formula shape, why multiplicative rather than additive, the class-base override path, and the `defaultMoveSpeed` fallback at `/Users/mmandrille/git/arpg-dev/server/internal/game/movement.go:18`. This prevents a future slice from silently breaking the formula's invariants.

**5. Add file doc comments to `fog_of_war.go` and `dungeon_room_layout.go`** [Low]
`/Users/mmandrille/git/arpg-dev/server/internal/game/fog_of_war.go` (no package-level doc) and `/Users/mmandrille/git/arpg-dev/server/internal/game/dungeon_room_layout.go` (no doc on `placeRoomLayout`) are navigated frequently during debugging. A two-sentence doc comment on each explaining the algorithm boundary (what it computes, what it does not) reduces the "why does fog interact with obstacles this way" investigation time. Low effort, high future payback.
