## Summary

The client improved materially since v53. Unknown server messages now warn, `_apply_delta` has
focused unit coverage, item and skill data load through static shared-data loaders, and Magic Bolt
presentation is no longer hardcoded into the panel/bar.

The thin-client boundary still holds. The main risk is cohesion: `main.gd` grew slightly to 4,543
LOC, still owns networking, world rendering, camera/input, UI orchestration, bot/replay hooks, boss
telegraphs, and skill casting. The next content manifest slice is feasible client-side if it stays
inside `SkillRulesLoader`, but broad item/class manifest work should wait for more UI/helper
extraction.

## 1. Architecture

- **[Strength] Shared-data singleton pattern is now in use for items and skills.**
  `client/scripts/item_rules_loader.gd:1` owns item rule/presentation loading, and
  `client/scripts/skill_rules_loader.gd:1` does the same for skills.
- **[Strength] Skill UI is data-backed.** `client/scripts/skill_bar.gd:175` resolves the first
  skill ID from `SkillRulesLoader`; `client/scripts/skills_panel.gd:313` resolves definitions and
  presentation metadata instead of hardcoding Magic Bolt labels.
- **[High] `main.gd` remains the client integration monolith.** At 4,543 LOC it still includes
  message dispatch (`client/scripts/main.gd:769`), delta application (`client/scripts/main.gd:878`),
  level/world rendering, all panel refresh, skill-cast helpers, boss phase visuals, and bot state.
  This is manageable for a skills-only manifest but risky for broader content-library rollout.
- **[Med] Panel/icon rendering duplication remains.** `_draw_item_icon` still exists separately in
  `inventory_panel.gd:492`, `shop_panel.gd:324`, `stash_panel.gd:335`, and
  `consumable_bar.gd:369`. Data loading was centralized, but presentation drawing still has four
  edit sites.

## 2. Technical

- **[Strength] Unknown message handling was fixed.** `_handle_message` now has a default arm with
  `push_warning` at `client/scripts/main.gd:785`, resolving a v53 protocol-debugging risk.
- **[Med] One malformed-delta direct access remains.** Most delta payload reads use `.get()`, but
  `equipped_update` still reads `c["slot"]` at `client/scripts/main.gd:914` and
  `client/scripts/main.gd:916`. Add a guard before using the slot.
- **[Strength] Delta/snapshot unit coverage exists.** `client/tests/test_delta_apply.gd:1` covers
  `_apply_delta` and `_apply_snapshot`, including malformed/unknown change cases at
  `client/tests/test_delta_apply.gd:173`.
- **[Med] Skill loader is permissive at runtime.** `SkillRulesLoader` silently returns empty maps if
  shared files are missing or malformed (`client/scripts/skill_rules_loader.gd:45` and
  `client/scripts/skill_rules_loader.gd:57`). CI validation catches committed data, but local
  runtime diagnostics should warn when the catalog is absent.

## 3. Maintainability

- **[Strength] v59 reduced skill UI duplication before adding more skills.** Shared skill rules and
  presentations are separate files, matching the content-library research direction.
- **[Med] New gameplay UI still requires several `main.gd` edits.** Panel construction, show/hide,
  refresh, and teardown are still coordinated through the main script. A future class/skill-tree UI
  will need a `UIOrchestrator` or similar seam.
- **[Med] Bot/replay scaffolding remains coupled to production state.** `bot_scenario_runner.gd`
  is 2,051 LOC and `main.gd` still exposes broad bot state. This is useful for CI, but it keeps
  test hooks close to production rendering and input paths.

## 4. Documentation

- **[Strength] Loader files now explain their purpose.** `SkillRulesLoader` states that mechanics
  stay separate from presentation metadata at `client/scripts/skill_rules_loader.gd:1`.
- **[Low] Panel files remain sparse on headers/docstrings.** This is not blocking, but richer
  comments around complex panel state would help future UI splits.
- **[Med] Manifest adoption checklist is not needed for the v60 review.** No client UI/art work was
  only if it touches client presentation beyond loader paths.

## Top 5 client refactors

1. Guard `equipped_update` slot reads in `_apply_delta`.
2. Keep the skills-only manifest confined to `SkillRulesLoader`; avoid broader `main.gd` edits.
3. Extract an `ItemIconRenderer` helper before item manifest splitting.
4. Add warning diagnostics when `SkillRulesLoader` cannot load committed shared files.
5. Plan a `UIOrchestrator` once class/skill-tree UI expands beyond the current two skill widgets.

*Evidence: direct reads of `main.gd`, `skill_rules_loader.gd`, `skill_bar.gd`, `skills_panel.gd`,
panel scripts, and `client/tests/test_delta_apply.gd`.*
