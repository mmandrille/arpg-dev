## Summary

The client still respects the thin-client boundary: v70 class rules are enforced by the server, and
the client consumes class/skill outcomes through snapshots and deltas. The skills content manifest
work from v60 is now visible in `SkillRulesLoader`, which emits useful warnings for missing or bad
manifest files.

The main client risk is still cohesion. `main.gd` grew from 4,543 LOC at v60 to 5,166 LOC and will
grow again in v71 for the create-character class picker unless UI orchestration is kept carefully
inside the character-select panel. One concrete malformed-delta hazard remains in `equipped_update`.

## 1. Architecture

- **[Strength] Thin-client authority remains intact.** The client does not decide class gates; the
  server rejects cross-class actions and the client renders the resulting state.
- **[Strength] Skill loading now uses the content manifest.** `SkillRulesLoader` loads the manifest
  once (`client/scripts/skill_rules_loader.gd:15`) and keeps mechanics/presentation separate.
- **[High] `main.gd` remains the integration monolith.** At 5,166 LOC it owns message dispatch,
  delta application, world rendering, panel refresh, skill helpers, menu flows, and bot hooks. v71
  should keep class picker logic in `character_select_panel.gd` and avoid broad main-script growth.
- **[Med] Item icon/panel drawing duplication remains.** The same item presentation concerns still
  span inventory, shop, stash, and consumable panels; item manifest work should wait for a shared
  renderer/helper.

## 2. Technical

- **[Strength] Unknown server messages are visible.** `_handle_message` warns on unknown types at
  `client/scripts/main.gd:802`, resolving the older silent-drop risk.
- **[Med] `equipped_update` still indexes `slot` directly.** `client/scripts/main.gd:935` and
  `client/scripts/main.gd:937` use `c["slot"]`; malformed server or test payloads can raise before
  a warning path. Guard with `str(c.get("slot", ""))` and skip empty slots.
- **[Strength] Manifest loader diagnostics improved.** Missing manifest nodes, duplicate IDs, bad
  paths, and parse failures call `push_warning` in `client/scripts/skill_rules_loader.gd:90`.
- **[Strength] Client CI coverage is broad.** `make ci` passed 30 client bot scenarios and the
  final headless smoke/unit suite, including skill panel/bar tests.

## 3. Maintainability

- **[Med] Bot hooks remain coupled to production state.** `bot_scenario_runner.gd` is 2,139 LOC and
  `main.gd` still exposes broad bot-facing control. This is valuable for CI, but new UI should add
  narrow bot methods rather than more global state.
- **[Med] v71 class picker is a cohesion risk.** The requested class blocks, tooltips, and class
  sprite in the hero list should live in the character-select panel and a small class icon helper,
  not in `main.gd`.
- **[Strength] Shared-data loaders are a good local pattern.** `SkillRulesLoader` is a clear model
  for future class presentation metadata once v71 adds UI sprites.

## 4. Documentation

- **[Strength] Client bot scenarios document important flows.** The scenario catalog now includes
  30 client scenarios.
- **[Low] Complex panels still have sparse local comments.** The code is readable, but panel state
  transitions would benefit from short boundary comments as class creation UI grows.

## Top 5 client refactors

1. Guard `equipped_update` slot access in `main.gd`.
2. Keep v71 class picker contained in `character_select_panel.gd` plus a small icon/control helper.
3. Add one narrow bot helper for selecting a class during character creation.
4. Extract item icon rendering before item/class presentation manifests expand.
5. Start a `UIOrchestrator` only when another panel flow would otherwise add main-script coupling.

*Evidence: direct reads of `main.gd`, `skill_rules_loader.gd`, panel scripts, client tests, v70 CI
output, and prior v60 client review; `make client-unit` and `make ci` passed.*
