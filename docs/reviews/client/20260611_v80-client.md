## Summary

The client still respects the authority boundary. v80 is a good example: `monster_aggro` is already
server-authored, and the client only maps it to floating threat text. Client bot debug state exposes
presentation without feeding gameplay back into the sim.

## 1. Architecture

[Strength] Existing event handling keeps gameplay facts on the wire and presentation local. Evidence:
`_apply_delta` routes server events at `client/scripts/main.gd:994`, and v80 maps `monster_aggro`
to display-only `AGGRO` text at `client/scripts/main.gd:1110`.

[Med] `main.gd` remains the largest client coordination file. It handles message parsing, event
routing, panels, combat text, camera, bot state, and replay helpers. This is workable, but future
UI slices should continue moving behavior into focused panel/helper scripts.

## 2. Technical

[Strength] Floating combat text has a stable bot-facing seam. Evidence: `_bot_damage_numbers`
returns text and variant at `client/scripts/main.gd:4886`, and the client bot runner already waits
for damage-number text/variant at `client/scripts/bot_scenario_runner.gd:233`.

[Low] Message payload access is mostly defensive, but `_handle_message` still directly indexes
`env["payload"]` for snapshot/delta/reject handling at `client/scripts/main.gd:798`. Server payloads
are controlled, yet malformed-delta tests have already justified defensive `.get()` patterns.

## 3. Maintainability

[Strength] The client bot runner has broad reusable step coverage, including event waits,
entity clicks, damage number assertions, panel assertions, and boss readability checks
(`client/scripts/bot_scenario_runner.gd:7`).

[Med] The bot runner's step registry is duplicated across wait/assert/action/all lists. That keeps
validation explicit, but new scenario verbs must update several places. A future small refactor
could derive `ALL_STEP_TYPES` from the three category lists.

## 4. Documentation

[Strength] Client-side visual work continues to record plugin decisions in plans and uses existing
in-repo presentation primitives instead of adopting dependencies for narrow display tasks.

## Top 5 client refactors

1. Extract combat event presentation from `main.gd` into a small event presenter/helper.
2. Derive client bot `ALL_STEP_TYPES` from the wait/assert/action arrays.
3. Use `.get("payload", {})` in `_handle_message` and add one malformed message unit test.
4. Keep adding client-bot scenarios for visible combat changes; they catch presentation regressions better than protocol bots.
5. Avoid widening `main.gd` when adding new panels; use focused scripts plus small `main.gd` wiring.
