## Summary

The client still respects the authority boundary. v80 is a good example: `monster_aggro` is a server-authored event, and the client only maps it into visible `AGGRO` feedback. Client bot state exposes presentation for tests without feeding gameplay decisions back into the sim.

Checks run for this refresh: `make client-unit` passed on the v80 baseline. Direct `make client-smoke` did not complete because it was run without a live local server; its server-independent unit phases passed, then the headless slice smoke failed at login with `not connected: 4`. A later full `make ci` attempt ran after unrelated uncommitted v81 skill changes appeared; it passed all 31 client bot scenarios, then failed in phase 9 because `client/tests/test_skill_rules_loader.gd:20` still expects 3 manifest-loaded skills while the dirty worktree loads 4.

## 1. Architecture

[Strength] Existing event handling keeps gameplay facts on the wire and presentation local. `_apply_delta` routes authoritative server events at `client/scripts/main.gd:994`, and v80 maps `monster_aggro` to display-only `AGGRO` text at `client/scripts/main.gd:1110`.

[Med] `client/scripts/main.gd` remains the largest client coordination file at 5297 lines. It handles message parsing, event routing, panels, combat text, camera, bot state, replay helpers, and debug output. This is workable, but future UI or combat-presentation slices should continue moving behavior into focused scripts.

## 2. Technical

[Strength] Floating combat text has a stable bot-facing seam. `_bot_damage_numbers` exposes text and variant at `client/scripts/main.gd:4886`, and the client bot runner waits on combat text through `_damage_number_matches` dispatch at `client/scripts/bot_scenario_runner.gd:233`.

[Low] Message payload access is mostly defensive, but `_handle_message` still directly indexes `env["payload"]` for snapshots, deltas, rejects, and errors at `client/scripts/main.gd:798`. Server payloads are controlled, yet malformed-delta tests have already justified defensive `.get()` patterns. A small test should pin malformed envelope behavior.

[Strength] Client bot coverage is now broad enough to protect visible combat work. `make/agents.mk:10` provides `make bot-client`, and `scripts/ci.sh:176` runs all client bot scenarios in CI.

## 3. Maintainability

[Med] The bot runner's step registry is duplicated across wait/assert/action/all lists. The explicit lists are readable, but every new scenario verb has to update several places. A future small refactor should derive `ALL_STEP_TYPES` from the category lists while keeping validation messages clear.

[Strength] The client uses focused panel scripts for many UI domains: shop, stash, skills, status effects, waypoint, boss health, and character panels are not all embedded directly in `main.gd`. Future slices should keep that direction.

## 4. Documentation

[Strength] Client-side visual work continues to record plugin decisions in plans and uses existing in-repo presentation primitives for narrow display tasks. The v80 as-built and review set keep the visual proof tied to the slice lifecycle.

## Top 5 client refactors

1. Extract combat event presentation from `main.gd` into a small event presenter/helper.
2. Derive client bot `ALL_STEP_TYPES` from wait/assert/action arrays.
3. Use `.get("payload", {})` in `_handle_message` and add one malformed envelope unit test.
4. Keep adding client-bot scenarios for visible combat changes; they catch presentation regressions better than protocol bots alone.
5. Avoid widening `main.gd` for new panels; use focused scripts plus small `main.gd` wiring.
