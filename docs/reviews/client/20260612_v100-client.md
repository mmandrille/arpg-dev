## Summary

The client still follows the thin-client model and recent rogue presentation work added useful visual feedback without moving gameplay authority client-side. The next gameplay slices should require little client logic at first: damage/resistance math belongs on the server, while the client may only need to display existing damage events and later support skeleton visuals.

The main client risk remains size. `main.gd` is the top-level coordinator for protocol application, HUD, panels, entity visuals, bot hooks, and combat text. New damage-type presentation should avoid adding a broad switch there unless the slice explicitly needs visible elemental effects.

## 1. Architecture

[Strength] The authority boundary is still written into the client entrypoint at `client/scripts/main.gd:1`: the client predicts movement and renders, while the server owns combat, loot, and inventory.

[Strength] Monster visuals are already cataloged and loaded through existing scene/script paths. A skeleton-style undead should reuse that asset/manifest pipeline instead of bypassing the catalog.

[Med] `client/scripts/main.gd:7` preloads many systems and remains a 6144-line coordinator. Any new client-facing damage-type colors or monster presentation should be kept in focused visual helpers where possible.

## 2. Technical

[Strength] The bot-facing damage-number hook now reports color data, which is useful for presentation assertions without requiring screenshots.

[Med] Damage types are not currently exposed as a distinct client presentation field. If the server starts including `damage_type` in events, the client should tolerate absence and render current default damage text unless a slice intentionally adds type-specific UI.

## 3. Maintainability

[Med] Client tests are present, but new monster visual coverage should stay headless and focused. Avoid scene-tree-heavy unit tests unless the behavior cannot be proven through bot visual/smoke paths.

[Low] Damage-type presentation can easily become color-coded-only feedback. If it is added, prefer event metadata plus small visual helpers so accessibility/localization work remains possible later.

## 4. Documentation

[Med] Future plans should name the exact visual verification command for undead if a bot scenario exists, for example `make bot-visual scenario=<scenario>`.

## Top 5 client refactors

1. Keep damage-type display optional and event-driven.
2. Add undead/skeleton visuals through the existing asset manifest and monster visual catalog.
3. Split new visual helpers out of `main.gd` when practical.
4. Use bot or client smoke proof for visible monster work.
5. Preserve graceful rendering when older events lack `damage_type`.

*Evidence: `client/scripts/main.gd:1`, `client/scripts/main.gd:7`, current file-size scan, existing monster scene preloads, and bot scenario coverage under `tools/bot/scenarios`.*
