# arpg-dev - Client (Godot 4 / GDScript) review at slice **v90**

**Date:** 2026-06-12
**Scope:** Godot thin client, event presentation, UI text, visual skill playback, client tests, scene/script organization.
**Baseline:** `main` @ `4a5afb5` (`chore: tune up lighting`); review pre-task cleaned CI blockers before this report landed.
**Stats:** Large client files: `client/scripts/main.gd` 5470 lines and `client/scripts/bot_scenario_runner.gd` 2120 lines.
**Overview:** [`../20260612_v90-overview.md`](../20260612_v90-overview.md)

## Summary

The client remains a presentation layer. Recent skill visuals improved in the right shape: projectile visuals, status markers, and skill replay behavior are split into focused scripts rather than packed only into `main.gd`.

The largest client gap is now localization readiness. Visible labels, menu copy, stat names, monster names, skill names, and future quest text need a single text service and catalog-backed lookup with English fallback.

## 1. Architecture

[Strength] Defensive message payload handling from v83 is in place. `client/scripts/main.gd` centralizes payload unwrap through `_envelope_payload`, reducing malformed-envelope crash risk.

[Strength] Recent presentation code is better split than older UI wiring. Projectile rendering lives in `client/scripts/projectile_visuals.gd`, and status-effect marker presentation lives in `client/scripts/player_status_effect_markers.gd`.

[Med] `client/scripts/main.gd` remains 5470 lines and still coordinates message parsing, event routing, HUD, menus, windows, bot state, and replay helpers. Localization should not add a new pile of string handling here; it should use a small focused text/locale service.

## 2. Technical

[Strength] Skill visual replay is now generic enough to support future skills. A single scenario can seed requested skill rank and `make skill-visual skill=all` can walk the data-driven catalog.

[Med] User-facing text is scattered. Settings, menus, skill labels/descriptions, stats, monster names, shop text, and future quest copy need stable text keys and a fallback path. Without that, Spanish support will duplicate literals and make future UI changes expensive.

[Low] Some current content identifiers have spelling drift, such as `ligthing`. Because ids may already be part of content/tool contracts, defer cleanup unless it is handled as a deliberate migration.

## 3. Maintainability

[Strength] Client bot coverage remains broad and is the right proof path for visible UI or presentation changes.

[Med] Localization must include tests that missing Spanish entries fall back to English. The client should not silently render raw keys except in a deliberate debug path.

## Top client recommendations

1. Add a focused text service script instead of embedding localization logic in `main.gd`.
2. Store English as the source catalog and let other locales fall back key-by-key.
3. Add Spanish via Settings selection after the English catalog path is proven.
4. Migrate touched UI panels and skill/monster presentation names through text keys.
5. Keep visual and localization helpers split into focused files by default.

*Evidence: direct reads of `client/scripts/main.gd`, `client/scripts/projectile_visuals.gd`, `client/scripts/player_status_effect_markers.gd`, client tests, and successful verification listed in the overview.*
