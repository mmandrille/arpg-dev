# arpg-dev - Client Review at slice **v110**

**Date:** 2026-06-13
**Scope:** Godot client, presentation boundaries, UI risk after v109 corpse recovery.
**Overview:** [`../20260613_v110-overview.md`](../20260613_v110-overview.md)

## Summary

The client still behaves as a renderer/input layer. v109 added corpse targeting, corpse labels, and
stash-style recovery affordances without making the client own item recovery. v110 did not add
client scope.

## Findings

[Strength] Corpse recovery presentation uses existing interactable and stash-panel patterns while
the server owns recovery outcomes.

[Strength] v110 avoided adding a premature upgrade UI, keeping repeat-upgrade semantics in the
authenticated HTTP/store layer first.

[Med] `client/scripts/main.gd` is 6294 lines and `client/tests/test_coop_client.gd` is 1305 lines.
Future market or elite-aura presentation should extract small helpers instead of adding broad
coordinator branches.

[Med] `client/scripts/character_select_panel.gd` now exceeds the new-file target and is baselined at
635 lines. Further character-status/death UI work should split formatting helpers.

[Low] Elite aura presentation can probably start with existing floating text or tint affordances.
Avoid introducing custom shader/VFX work until the server aura payload is stable.

## Recommendations

1. Keep v111 market purchase UI out of scope unless store/HTTP proof is already complete.
2. If client market affordances are added later, isolate them in a market panel script.
3. For elite aura, first prove server behavior and bot assertions; add visual polish only after.
4. Treat any new `main.gd` growth as a prompt to extract presenter helpers.
