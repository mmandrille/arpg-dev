# v120 Client Review

Date: 2026-06-13
Scope: Godot client, panels, client bot coverage, v111-v120 UI additions

## Summary

The client stayed thin across the batch. Market and blacksmith work added player-facing panels while
leaving mutation authority in HTTP/server routes. The biggest client-side risk is still
`client/scripts/main.gd`, which coordinates every service panel and continues to absorb glue code.

## Strengths

- Market board, purchase, active-offer, and blacksmith upgrade flows use focused panels and existing
  server routes.
- The blacksmith NPC gives the upgrade mechanic an in-world town entrypoint instead of hiding it in
  debug or backend-only surfaces.
- v120 made the skills-panel unit test derive Magic Bolt requirements, mana cost, and max rank from
  `SkillRulesLoader`, reducing balance-tuning friction without changing production UI behavior.
- Bot scenarios now cover the recent player-facing market, blacksmith, elite preview, and unique
  drop flows.

## Findings

### Medium: `main.gd` still owns too much service-panel coordination

`client/scripts/main.gd` is about 6.4k lines. The service bridge area now includes shop, stash,
corpse, bishop, market, and blacksmith open/configuration paths. The panels are focused, but their
lifecycle coordination still lands in the same central script.

Recommendation: before adding another town NPC or service panel, extract a small
town-service/panel coordinator that owns open/close refresh behavior and delegates to the specific
panel.

### Medium: client bot scripting remains centralized

`client/scripts/bot_scenario_runner.gd` and `tools/bot/run.py` are still broad action/assertion
surfaces. The v84 registry cleanup helped, but scenario growth will keep increasing review cost
unless action groups move behind small domain registries.

Recommendation: split future market/item/skill bot actions into focused registries while preserving
the current scenario JSON contract.

### Low: UI tests should keep separating behavior from tuning

The skills-panel test now avoids copied Magic Bolt tuning values. Similar tests may still pin
balance through exact values where the UI only needs requirement, affordability, or state-change
proofs.

Recommendation: keep exact text/layout checks where they verify presentation contracts, but derive
gameplay values from shared loaders when the number is only fixture setup.
