## Summary

The client preserved the thin-client boundary while adding meaningful player-facing work: inventory
items can be staged into market and blacksmith flows, Ranger has class and skill presentation, and
market active-offer UI received correction work after the accepted-offer stash delivery fix. v127
also acted on the prior review by extracting a first town-service bridge from `main.gd`.

## 1. Architecture

[Strength] `TownServiceBridge` is a useful first extraction. It owns market/blacksmith inventory
context toggles and inventory stage routing without taking authority over item mutations
(`client/scripts/town_service_bridge.gd:5`, `client/scripts/town_service_bridge.gd:40`).

[Strength] Inventory-origin market/blacksmith flows remain client staging only. The client can show
context and send intents, while ownership changes still go through HTTP/server routes.

[Med] `main.gd` is still the presentation and world coordinator at 6663 lines. v127 removed one
pressure point, but ground-loot presentation, panel lifecycle, world entities, combat cues, and many
service bridges still share one script. Future unique-test-chest work should avoid adding chest UI
and loot generation directly into `main.gd` if a focused helper can own the presentation.

## 2. Technical

[Strength] Rarity presentation already has a unique path for ground equipment tinting, so the next
unique-test surface can reuse existing visual language rather than inventing a new item renderer
(`client/scripts/main.gd:5255`).

[Med] Bot/client scenario support is effective but centralized. `bot_scenario_runner.gd` has grown
to 2354 lines, and the Python runner is still the main domain-action surface. New chest assertions
or pickup helpers should be added as narrow reusable steps rather than scenario-specific special
cases.

[Low] The client still lacks unique-specific inspection UI. That was an explicit v119 deferral, and
it matters more now that unique testing is becoming player-facing.

## 3. Maintainability

[Strength] v127 added `client/tests/test_town_service_bridge.gd`, proving the extraction with
headless coverage instead of relying on manual town interaction.

[Med] The town service extraction should continue incrementally. Current bridge methods are static
and narrow; a future slice could move service open/close refresh orchestration out of `main.gd`
without changing panel internals.

[Low] Client presentation uses code-native placeholder VFX and deterministic generated assets. That
is still appropriate for gameplay iteration, but production unique art/audio remains deferred.

## 4. Documentation

[Strength] Recent as-built docs include manual visual verification for Ranger Volley:
`make bot-visual scenario=60_ranger_volley_and_visual_showcase`.

[Med] The next unique chest slice should name its visual scenario command if it creates a visible
town object. The existing review/AGENTS policy asks for exact bot-visual commands when visual
effects have a scenario.

## Top 5 client refactors

1. Keep expanding `TownServiceBridge` only around service coordination; do not let it become another broad coordinator.
2. Put unique-test-chest presentation in a focused script/helper if it needs custom behavior.
3. Add a small unique item/effect inspection view before relying on names/effects hidden in raw inventory rows.
4. Split bot scenario runner assertions by domain when adding new unique/chest assertions.
5. Preserve server-authored outcomes for all chest contents; the client should only render and request interaction.
