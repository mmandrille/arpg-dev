## Summary

The client gained useful player-facing proof for the unique batch without taking authority away from
the server. Unique inspection, market purchase/offer UI, blacksmith UI, and the purple chest client
proof all run through server-backed state and the CI client bot suite. The main client risk remains
coordinator size: `main.gd` still owns world presentation, many UI panel bridges, bot hooks, and
loot presentation.

## 1. Architecture

[Strength] The thin-client boundary is still explicit at the entrypoint: `main.gd` describes itself
as renderer/prediction over the authoritative server, with combat, loot, and inventory outcomes
server-owned (`client/scripts/main.gd:1`).

[Strength] Client market flows remain panel-driven UI over server state. Bot hooks in `main.gd`
delegate publish, purchase, view-offers, and accept-offer actions into `market_panel.gd` instead of
mutating inventory locally (`client/scripts/main.gd:6464`, `client/scripts/main.gd:6474`,
`client/scripts/main.gd:6479`, `client/scripts/main.gd:6484`).

[Med] `main.gd` is still a broad coordinator. It preloads dozens of systems and panels
(`client/scripts/main.gd:7` through `client/scripts/main.gd:45`), owns loot presentation helpers
(`client/scripts/main.gd:5270`), status-effect tint routing (`client/scripts/main.gd:5210`), and
bot delegation hooks (`client/scripts/main.gd:6405` through `client/scripts/main.gd:6541`). Future
presentation slices should continue extracting helpers instead of adding more branches here.

## 2. Technical

[Strength] Ground loot presentation can now show real item models when available and falls back to
rarity-aware primitives when not (`client/scripts/main.gd:5270`, `client/scripts/main.gd:5335`).
That is enough for current unique chest proof without client-authored item creation.

[Strength] The client bot suite is a meaningful regression gate. The v139 `make ci` run passed 40
Godot client bot scenarios, including market board, market purchase, active offers, blacksmith
upgrade, and unique chest proof.

[Med] `bot_scenario_runner.gd` has become the client-side DSL registry and validator. Step types,
assertions, actions, and static validation all live in one 2376-line file
(`client/scripts/bot_scenario_runner.gd:7`, `client/scripts/bot_scenario_runner.gd:20`,
`client/scripts/bot_scenario_runner.gd:46`, `client/scripts/bot_scenario_runner.gd:2130`). The
next client-bot expansion should split domain validators or assertion handlers.

## 3. Maintainability

[Strength] UI panel ownership is better than earlier reviews: shop, stash, market, blacksmith,
skills, status effects, and character panels have dedicated scripts and tests listed in the
CODEMAP (`docs/CODEMAP.md:12`, `docs/CODEMAP.md:13`, `docs/CODEMAP.md:19`).

[Med] Bot-facing methods in `main.gd` are useful for tests but now form a long adapter section
(`client/scripts/main.gd:6405` through `client/scripts/main.gd:6541`). A small bot facade could
keep scenario compatibility while reducing direct test hooks in the main scene.

[Low] Client presentation still uses generated/placeholder assets for much of the game. That is
acceptable for the current active-development phase; production art remains a product scope item,
not a correctness blocker.

## 4. Documentation

[Strength] Recent as-built docs name client visual proof where relevant, including the unique chest
client proof and the CI-backed client bot scenarios.

[Med] The next visible presentation slice should keep naming exact `make bot-visual scenario=...`
commands in the spec/plan/as-built when visual review matters.

## Top 5 client refactors

1. Extract a bot facade from `main.gd` so scenario hooks stop growing the scene coordinator.
2. Split `bot_scenario_runner.gd` by step domain or validation domain.
3. Continue moving service-specific UI coordination into focused bridges, following
   `town_service_bridge.gd`.
4. Keep unique/market/blacksmith panels as server-state renderers, not local item mutators.
5. Add visual scenario commands to future specs whenever a visible client feature changes.
