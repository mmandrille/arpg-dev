## Summary

The client directly addressed the two v140 architectural recommendations. `main.gd` now keeps public bot wrappers but delegates adapter bodies to `BotFacade`, and `BotScenarioRunner` now delegates step catalog/validation, wait handlers, assertion handlers, and action queueing. The thin-client boundary remains explicit. The next client risk is ordinary presentation concentration: `main.gd` still owns scene assembly and many UI/presentation routes, and the bot runner's validation/assertion surfaces are smaller but still large enough to split further by domain.

## 1. Architecture

[Strength] The authoritative boundary remains visible at the entrypoint: `main.gd` identifies itself as a renderer/prediction layer, with combat, loot, and inventory outcomes owned by the server (`client/scripts/main.gd:1`).

[Strength] Bot adapter bodies are no longer embedded in the scene coordinator. `main.gd` wrappers call `BotFacade` for shop, stash, market, blacksmith, hotbar, stat, and skill actions (`client/scripts/main.gd:6399`, `client/scripts/main.gd:6411`, `client/scripts/main.gd:6431`, `client/scripts/main.gd:6442`, `client/scripts/main.gd:6457`, `client/scripts/main.gd:6465`, `client/scripts/main.gd:6473`). `BotFacade` routes those calls to focused panels without mutating game state locally (`client/scripts/bot_facade.gd:5`, `client/scripts/bot_facade.gd:23`, `client/scripts/bot_facade.gd:71`, `client/scripts/bot_facade.gd:101`, `client/scripts/bot_facade.gd:125`).

[Strength] Client bot execution now has explicit runner submodules. The runner preloads the step catalog, wait handlers, assertion handlers, and action handlers (`client/scripts/bot_scenario_runner.gd:7`), dispatches wait/assert/action categories (`client/scripts/bot_scenario_runner.gd:98`, `client/scripts/bot_scenario_runner.gd:105`, `client/scripts/bot_scenario_runner.gd:110`), and delegates implementation (`client/scripts/bot_scenario_runner.gd:123`).

[Med] `main.gd` still owns too many presentation domains. It preloads scene systems and panels in one file (`client/scripts/main.gd:7` through `client/scripts/main.gd:45`), and remains the central place where new visual/UI features are likely to attach.

## 2. Technical

[Strength] The bot facade has headless unit coverage with fake panels. The test proves representative panel routing, hotbar routing, skill selection, direct skill casts, and directional casts without requiring scene-tree children (`client/tests/test_bot_facade.gd:143`, `client/tests/test_bot_facade.gd:154`, `client/tests/test_bot_facade.gd:184`).

[Strength] The client bot DSL has a stronger structure after v144. Step types and static validation live in `BotStepCatalog` (`client/scripts/bot_step_catalog.gd:5`, `client/scripts/bot_step_catalog.gd:18`, `client/scripts/bot_step_catalog.gd:44`, `client/scripts/bot_step_catalog.gd:64`), wait dispatch lives in `BotWaitHandlers` (`client/scripts/bot_wait_handlers.gd:5`), assertion dispatch lives in `BotAssertionHandlers` (`client/scripts/bot_assertion_handlers.gd:5`), and action queuing is tiny (`client/scripts/bot_action_handlers.gd:5`).

[Med] Bot assertion and validation chains remain domain-heavy. `BotStepCatalog.validate_step` still carries detailed requirements for many unrelated UI/gameplay domains (`client/scripts/bot_step_catalog.gd:85`), and `BotAssertionHandlers.evaluate` is still one large match across panels, progression, combat feedback, market, bishop, blacksmith, boss UI, and multiplayer state (`client/scripts/bot_assertion_handlers.gd:5`).

## 3. Maintainability

[Strength] The trend is down: `main.gd` 6769 -> 6703, and `bot_scenario_runner.gd` 2376 -> 1665 since v140.

[Med] `inventory_panel.gd` remains 1559 lines and is a likely future hotspot because inventory, stash, market, requirements, and comparison UI all touch nearby concepts. New inventory-facing UI should prefer panel-specific helpers rather than growing this file.

[Low] Placeholder/generated visuals are still acceptable for this stage. The important boundary is that the client presents server-authored state rather than authoring outcomes.

## 4. Documentation

[Strength] CODEMAP now lists `bot_facade.gd`, `bot_step_catalog.gd`, `bot_wait_handlers.gd`, `bot_assertion_handlers.gd`, and `bot_action_handlers.gd` under Bot / scenarios, so future client-bot work starts in focused files (`docs/CODEMAP.md:24`).

[Low] Future visible client slices should continue naming the exact `make bot-visual scenario=...` command when visual review matters.

## Top 5 client refactors

1. Split `BotAssertionHandlers` by UI/gameplay domain when the next client-bot assertion grows.
2. Split `BotStepCatalog.validate_step` by step family or domain before adding more validation cases.
3. Extract another `main.gd` presentation domain only when a slice already touches that domain.
4. Keep bot facade wrappers thin and panel-owned; do not put panel mutation logic back into `main.gd`.
5. Watch `inventory_panel.gd` during market/stash/equipment slices and split helper modules when a cohesive subdomain emerges.
