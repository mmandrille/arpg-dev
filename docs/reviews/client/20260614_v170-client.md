## Summary

The client batch stayed within the thin-client boundary and paid down the exact client hotspots flagged at v160. Inventory transfer routing now delegates to `InventoryTransferRouter`, UI/menu assertions have a focused helper, and action-step validation moved out of `BotStepCatalog`. The remaining client risk is still concentration in `main.gd`, `inventory_panel.gd`, and `bot_scenario_runner.gd`.

## 1. Architecture

[Strength] Inventory transfer routing is no longer entirely embedded in `inventory_panel.gd`. Double-click, shift-click, drop, and slot-kind decisions delegate through `InventoryTransferRouterScript` (`client/scripts/inventory_panel.gd:850`, `client/scripts/inventory_panel.gd:889`).

[Strength] Client bot UI/menu assertion ownership is now split. `BotAssertionHandlers.evaluate` delegates first to `BotUiAssertionHandlers.try_evaluate` (`client/scripts/bot_assertion_handlers.gd:7`), and the helper owns menu, character panel, session, multiplayer filter, settings, pause, and character-info assertions (`client/scripts/bot_ui_assertion_handlers.gd:5`).

[Med] `main.gd` remains the largest client coordinator at 6724 lines. No v161-v170 slice worsened the authority boundary, but future visible features should avoid routing more panel or scenario helper logic through the main scene by default.

## 2. Technical

[Strength] Action-step validation now has a focused helper. `BotStepCatalog.validate_step` delegates action-family validation to `BotActionStepValidator.validate` before handling waits/assertions (`client/scripts/bot_step_catalog.gd:88`), and the helper owns key, click, drag, stash, multiplayer, market, blacksmith, and waypoint action field checks (`client/scripts/bot_action_step_validator.gd:23`).

[Strength] The client still acts as renderer/input layer. Recent objective chest presentation, session filtering, inventory routing, and assertion splits display or validate state; they do not make client-side gameplay outcomes authoritative.

[Med] `bot_scenario_runner.gd` grew to 1687 lines. The assertion and validation helper splits reduce some pressure, but scenario execution, state capture, UI interactions, and failure reporting still share one runner.

## 3. Maintainability

[Strength] The v160 client recommendations were substantially addressed. `inventory_panel.gd` dropped from 1583 to 1528 lines, `bot_assertion_handlers.gd` dropped from 298/299 to 267 lines, and `bot_step_catalog.gd` dropped from 312/322 to 250 lines.

[Med] `inventory_panel.gd` still owns equip eligibility and preferred slot rules around `client/scripts/inventory_panel.gd:902`. A future inventory/equipment slice could move those pure routing helpers into the router if it needs to touch them.

[Low] Client scenario coverage is strong enough that additions should stay selective. CI now reports 41 client bot scenarios; each new visual proof should have a clear player-visible contract and a named scenario handoff.

## 4. Documentation

[Strength] v165, v166, and v168 docs record the client helper splits and focused verification.

[Low] CODEMAP can keep pointing at the existing client domains for now, but any extraction out of `main.gd` or `bot_scenario_runner.gd` should update it so agents start from the smaller file.

## Top 5 client refactors

1. Extract one `main.gd` presentation or panel-bridge domain when a visible feature already touches it.
2. Move remaining pure equip-slot eligibility helpers from `inventory_panel.gd` into the transfer router during the next equipment/inventory slice.
3. Split `bot_scenario_runner.gd` by state capture, UI action execution, or assertion plumbing when scenario work touches those sections.
4. Continue splitting `BotAssertionHandlers` by gameplay domain, especially shop/stash/market assertions.
5. Keep all client-side additions display-only or input-only; server state must remain authoritative.
