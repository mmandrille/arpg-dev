## What improved

- Quest journal UI is backed by focused panel and bot assertions. The client scenario `tools/bot/scenarios/client/43_quest_journal_foundation.json:21` waits for the journal, and `client/scripts/bot_assertion_handlers.gd:137` routes journal assertions through a dedicated assertion helper.
- Elite objective presentation now has both HUD and minimap proof. The client scenarios `tools/bot/scenarios/client/44_elite_objective_hud.json:20` and `tools/bot/scenarios/client/45_elite_objective_minimap_pin.json:20` wait on the expected UI state.
- The main client owns sync points for quest and elite objective state in `client/scripts/main.gd:5950`, `client/scripts/main.gd:5969`, and `client/scripts/main.gd:5974`. This is clear, but it is also the next extraction target if those systems grow.
- Client bot coverage remains broad: `make ci` passed 45 client scenarios, including recent quest, elite objective, market, blacksmith, and chest presentation flows.

## Risks

1. `client/scripts/main.gd` is still the primary client risk at 6,383 lines. It owns panel construction, sync, bot debug state, interactable presentation, loot labels, blacksmith actions, quest journal state, and elite-objective state. New UI slices should extract a nearby sync/debug/presentation helper before adding more branches.

2. `client/scripts/bot_scenario_runner.gd` is 1,690 lines and close to its ratchet allowance. v171-v180 wisely used helper assertion files for quest and elite objective checks, but the runner still owns a large amount of assertion dispatch and failure text.

3. Client feature additions depend on debug-state shape. The scenario system is useful, but adding fields to `main.gd` debug state can become an implicit API if assertions grow without a typed facade.

4. Inventory and blacksmith flows remain coupled through main-panel routing. That is acceptable for current scale, but upgrade-resource spending will likely need a more explicit client-side resource display or panel state contract.

## Recommended client work

- For the next quest or elite objective slice, extract quest/elite sync and debug-state assembly out of `main.gd` before adding new UI behavior.
- For the next client bot assertion type, route it through `bot_assertion_handlers.gd` and focused helper scripts rather than growing the central runner.
- If upgrade shards become spendable, add a client bot scenario that proves inventory/resource display and failed-spend messaging, not just protocol pickup.
