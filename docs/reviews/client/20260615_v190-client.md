## Summary

The client remains a renderer/input layer and did not take ownership of gameplay outcomes in v181-v190. It renders server-owned state for set items, companion/monster entities, elite minions, rare affix outcomes, and Paladin defensive effects. The client bot suite remains a major strength: all 45 scenarios passed in `make ci`.

The main client risk is not correctness today; it is continued accumulation in `main.gd`, especially around status-effect presentation and bot debug-state assembly.

## 1. Architecture

- **[Strength] Presentation stays downstream from protocol state/events.** Holy Shield and Sanctuary presentation are driven by `skill_effect_started` / `skill_effect_ended` events in `main.gd` (`client/scripts/main.gd:1094`, `client/scripts/main.gd:1123`, `client/scripts/main.gd:1132`).
- **[Strength] The status-effect marker helper is a meaningful extraction.** `player_status_effect_markers.gd` owns marker names, sync helpers, pulse creation, and debug counters instead of adding all visual-node construction directly to `main.gd` (`client/scripts/player_status_effect_markers.gd:23`, `client/scripts/player_status_effect_markers.gd:187`, `client/scripts/player_status_effect_markers.gd:526`).
- **[Med] `main.gd` still owns too many presentation sync seams.** It handles event routing, status-effect sync, area pulse targeting, quest/elite objective sync, and bot debug snapshots in one file (`client/scripts/main.gd:1678`, `client/scripts/main.gd:5985`, `client/scripts/main.gd:6000`). The next UI/status slice should extract a focused presentation-sync facade.

## 2. Technical

- **[Strength] Holy Shield visual radius follows shared skill data.** `_holy_shield_aura_radius()` reads the skill definition and uses the configured effect radius (`client/scripts/main.gd:1716`), matching the shared rule at `shared/rules/skills.v0.json:520`.
- **[Med] Sanctuary dome radius is hardcoded in client presentation.** The Sanctuary rule has `radius: 5` in shared data (`shared/rules/skills.v0.json:584`), but the client dome mesh hardcodes `5.0` for both sphere and ground radii (`client/scripts/player_status_effect_markers.gd:247`, `client/scripts/player_status_effect_markers.gd:260`). If tuning changes, the visual can silently drift from server authority.
- **[Strength] Bot-facing debug state is rich enough to prove recent UI effects.** Local player and entity debug snapshots expose Holy Shield/Sanctuary markers and pulse counts (`client/scripts/main.gd:5954`, `client/scripts/main.gd:6019`), and smoke includes the status-effect presentation test.

## 3. Maintainability

- **[Med] Client bot and presentation helpers are better than before, but hotspots remain near ratchet limits.** `client/scripts/bot_scenario_runner.gd` is 1,640 lines and should keep routing new assertion families through helper scripts. `client/scripts/main.gd` is 6,376 lines and should not absorb the next quest/status/minimap/debug domain.
- **[Low] `player_status_effect_markers.gd` is useful but close to becoming a visual-effect bucket.** It already owns Rage, Burning, Elite Command, Pinning Root, Holy Shield, and Sanctuary markers. The next new effect family should either split by effect category or introduce a small registry.

## 4. Documentation

- **[Strength] The stale Godot plugin research requirement was removed before this review.** The active workflow now tells agents to check in-repo Godot code/assets and record an adopt/borrow/reject decision rather than requiring a deleted file.
- **[Low] Client visual scenarios are strong, but visual-only changes still need explicit scenario names in slice handoffs.** The v190 visual proof should continue to name focused commands such as `make bot-visual scenario=paladin_sanctuary` when future visual work changes status effects.

## Top 5 client refactors

1. Make Sanctuary presentation read the effect radius from `SkillRulesLoader`, matching the Holy Shield pattern.
2. Extract status-effect event routing/debug-state assembly from `client/scripts/main.gd`.
3. Keep adding client bot assertion/action families through focused helper scripts rather than growing `bot_scenario_runner.gd`.
4. Split `player_status_effect_markers.gd` by effect family if the next slice adds more VFX marker types.
5. Preserve thin-client authority by proving new visible outcomes through protocol state/events before adding presentation.
