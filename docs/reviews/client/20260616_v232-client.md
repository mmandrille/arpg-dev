## Summary

The client remains a renderer/input layer at v232. The latest queue added code-native audio,
wallet display, market cancellation, and mercenary recovery controls without moving combat, item,
economy, or companion authority into GDScript. Current `make client-unit` passed. Standalone
`make client-smoke` was not a useful isolated proof because its live-login gate expects a server
already listening on `localhost:18081`; the full v226-v232 `make ci` covered live smoke and all
49 client bot scenarios.

Maintainability is stable rather than improved: `main.gd` stayed at 5,760 lines, while new UI went
mostly into focused panel/bridge/controller files. `market_panel.gd` remains above target and grew
with cancel-offer behavior.

## 1. Architecture

- **[Strength] Thin-client boundary is explicit and still respected.** `main.gd` documents server
  ownership of combat, loot, and inventory outcomes (`client/scripts/main.gd:1`), and recent UI
  code emits intents or renders server state instead of computing outcomes locally.
- **[Strength] Audio behavior is presentation-local.** `ClientAudioController` owns generated cue,
  boss music, and ambience streams (`client/scripts/client_audio_controller.gd:49`,
  `client/scripts/client_audio_controller.gd:85`, `client/scripts/client_audio_controller.gd:115`),
  with `ClientAudioBridge` providing narrow calls from the coordinator (`client/scripts/client_audio_bridge.gd:69`,
  `client/scripts/client_audio_bridge.gd:99`, `client/scripts/client_audio_bridge.gd:109`).
- **[Strength] Market/mercenary UI features remain intent/state adapters.** Market cancel emits a
  cancel action only (`client/scripts/market_panel.gd:201`, `client/scripts/market_panel.gd:598`);
  mercenary recovery renders roster/loss state from events and companion rows (`client/scripts/mercenary_panel.gd:83`,
  `client/scripts/mercenary_panel.gd:94`, `client/scripts/mercenary_panel.gd:121`).
- **[Med] `main.gd` remains the broad presentation router.** It still preloads most feature panels
  and controllers (`client/scripts/main.gd:21`, `client/scripts/main.gd:45`, `client/scripts/main.gd:50`)
  and owns entity reaction/projectile/bot facade routing (`client/scripts/main.gd:1702`,
  `client/scripts/main.gd:5076`, `client/scripts/main.gd:5512`).

## 2. Technical

- **[Strength] Client checks are green where rerun in isolation.** `make client-unit` passed across
  Godot asset import, golden tests, panel tests, audio tests, and delta/loot-filter tests.
- **[Strength] Client bot coverage expanded.** Client bot scenarios increased to 49, including
  market cancel and mercenary recovery proofs.
- **[Low] Standalone live smoke needs an explicit server harness.** `make client-smoke` failed only
  at the final live-login gate with `not connected: 4`; that target expects the test server that
  `make ci` provides.

## 3. Maintainability

- **[Strength] New audio code is extracted.** Audio cue generation and settings synchronization live
  in `client_audio_controller.gd` and `client_audio_bridge.gd`, avoiding more direct `main.gd`
  growth.
- **[Med] Large client files remain managed debt.** The ratchet still grandfathers `main.gd`,
  `inventory_panel.gd`, `market_panel.gd`, and several client tests (`.maintainability/file-size-baseline.tsv:7`,
  `.maintainability/file-size-baseline.tsv:8`, `.maintainability/file-size-baseline.tsv:14`).
- **[Low] Market panel is now a stronger extraction candidate.** It owns browse, publish, offer,
  and cancel UI in one 1,078-line file; future market UI work should split rows/actions rather
  than adding another behavior cluster.

## 4. Documentation

- **[Strength] Visual proof remains named.** v226-v232 as-built files name the runnable
  `make bot-visual scenario=...` commands for audio, wallet, market, and mercenary UI inspection.
- **[Med] Agent process docs disagree on review/refactor order.** `PROGRESS.md` says write
  `$review` before `$refactor` (`PROGRESS.md:39`), while other entrypoint docs still say
  `$refactor` before `$review`. That can make future agents choose the wrong baseline.

## Top 5 client refactors

1. Fix agent process docs so review/refactor order is consistent.
2. Extract the next touched `main.gd` routing path, prioritizing bot facade, entity reaction, or
   projectile presentation.
3. Split future market UI work out of `market_panel.gd` instead of growing it further.
4. Keep audio/gameplay boundaries strict: audio cues should consume events/settings only.
5. Keep client smoke expectations clear: standalone target needs a running server, while `make ci`
   owns the full live harness.
