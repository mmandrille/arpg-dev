## Summary

The client still behaves as a renderer, input layer, and presentation/debug surface. Recent work improved fog visuals, minimap state, performance status, monster and boss presentation, imported class/monster rigs, and a model viewer. Server-owned gameplay facts still originate from Go snapshots, deltas, events, and shared rule/asset metadata.

The client is also where the review-time risk appeared: the broad Godot client bot step failed two scenarios (`mercenary_recovery_ui`, `door_fog_toggle`) in the official `make ci` run. Both passed when rerun individually, so this looks timing/order sensitive rather than a deterministic functional break, but it is still a cadence-gate failure and should be paid down before more feature slices.

## 1. Architecture

- **[Strength] Fog/minimap presentation remains downstream of existing state.** `FogOfWarOverlay` consumes progression, walls, and supplied occluders (`client/scripts/fog_of_war_overlay.gd:133`, `client/scripts/fog_of_war_overlay.gd:145`, `client/scripts/fog_of_war_overlay.gd:156`) and exposes bot-readable debug state (`client/scripts/fog_of_war_overlay.gd:167`). `DiscoveryMinimapState` derives explored cells, objective pins, and quest-path direction from current client entities (`client/scripts/discovery_minimap_state.gd:25`, `client/scripts/discovery_minimap_state.gd:138`).
- **[Strength] New attack presentation is event-driven.** The client plays `dive` or `pounce` only when server combat events include `attack_style`, and it looks up the source monster controller rather than deciding combat locally (`client/scripts/monster_attack_animation_events.gd:3`).
- **[Strength] Boss telegraph presentation consumes server metadata.** `BossVisualsController` applies server `boss_phase_started` fields (`client/scripts/boss_visuals_controller.gd:107`) and maps `line`, `cone`, `melee_contact`, and `rectangle` hit shapes to code-native meshes (`client/scripts/boss_visuals_controller.gd:221`, `client/scripts/boss_visuals_controller.gd:232`).
- **[Strength] Model viewer is isolated tooling.** `make model` validates the asset id through Python before launching Godot (`make/client.mk:29`), and the Godot scene loads catalog data without touching production gameplay flow (`client/scripts/model_viewer.gd:42`, `client/scripts/model_viewer.gd:77`).

## 2. Technical

- **[Med] Full client-bot sweep failed two timing-sensitive scenarios.** In review-time `make ci`, `mercenary_recovery_ui` timed out waiting for the lost-state panel update after player death, and `door_fog_toggle` expected a reclosed door extra occluder but saw `extra_occluder_count=0`. Both focused reruns passed with `HEADLESS=1 SCENARIO=49_mercenary_recovery_ui ./scripts/bot_client_local.sh` and `HEADLESS=1 SCENARIO=73_door_fog_toggle ./scripts/bot_client_local.sh`, so the next refactor pass should harden waits/state synchronization instead of treating the failures as closed.
- **[Strength] Client unit/smoke coverage is broad.** The review-time Godot smoke passed, including golden, item visual, projectile visual, rig, animation, model viewer, bot facade, co-op, minimap, panels, status, net client, and headless slice smoke gates (`scripts/client_smoke.sh:92`, `scripts/client_smoke.sh:103`, `scripts/client_smoke.sh:104`, `scripts/client_smoke.sh:120`).
- **[Strength] Model viewer has both Python and Godot coverage.** Python tests prove catalog grouping and equipment exclusion (`tools/assets/test_model_catalog.py:90`, `tools/assets/test_model_catalog.py:112`), and GDScript tests load a character and monster asset through the viewer (`client/tests/test_model_viewer.gd:10`).

## 3. Maintainability

- **[Strength] New model-viewer code is focused.** `client/scripts/model_viewer.gd` is 280 lines and `tools/assets/model_catalog.py` is 151 lines, keeping the v284 tool outside `main.gd`.
- **[Med] Large client coordinators remain the dominant cost.** `main.gd`, `bot_scenario_runner.gd`, `inventory_panel.gd`, `shop_panel.gd`, `stash_panel.gd`, `skills_panel.gd`, and large client tests are still grandfathered. Review-time file-size ratchet passed but reported 34 grandfathered files / 64,006 lines.
- **[Low] The model catalog exists twice.** Python (`tools/assets/model_catalog.py:33`) and GDScript (`client/scripts/model_viewer.gd:77`) implement parallel catalog discovery from the same JSON sources. Current tests cover representative assets, but a shared generated catalog or parity test would reduce future drift as more previewable asset kinds are added.

## 4. Documentation

- **[Strength] Visual proof commands are consistently recorded.** Recent as-built notes name focused `make bot-visual` or `make model` commands for fog, minimap, monster visuals, boss patterns, and model viewer work.
- **[Med] CI documentation should now mention review-time client-bot flakes.** `docs/as-built/v284_model-viewer-tool.md:44` already records one transient `mercenary_recovery_ui` failure during v284. The review found the same scenario plus `door_fog_toggle` in the broad client-bot sweep, so the next maintenance pass should make these scenarios less timing-sensitive.

## Top 5 client refactors

1. Harden `mercenary_recovery_ui` and `door_fog_toggle` waits/state proofs so full-suite order/timing does not create false red CI.
2. Add a catalog parity check or generated shared catalog for Python `model_catalog.py` and Godot `ModelViewer.catalog_rows()`.
3. Split the next touched `main.gd` debug/presentation sync path instead of adding more inline state keys.
4. Keep future fog/minimap polish in focused classes and client bot debug assertions rather than growing `main.gd`.
5. Keep production monster/boss VFX and artist-quality rig deformation as planned visual slices, not opportunistic refactors.
