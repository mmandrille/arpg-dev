## Summary

The client made real progress in feel without violating the thin-client boundary. The batch added
focused helpers for short attack buffering, sticky target memory, local windup/recovery
presentation, movement smoothing, command retarget grace, and melee lunge. Each helper has focused
unit or scenario coverage, and bot state now exposes the presentation/debug surfaces needed by the
85-scenario client sweep.

The top client risk is no longer the v284 broad-suite flake. It is the accumulation of input and
presentation orchestration in `main.gd` plus several code-owned feel constants. The specs documented
code ownership for the microconstants, which was acceptable for a fast batch, but the next tuning
pass should centralize that policy in data or a dedicated presentation config owner.

## 1. Architecture

- **[Strength] Feel helpers are focused and client-only.** `CombatInputBuffer` stores one
  short-lived monster target with liveness guards (`client/scripts/combat_input_buffer.gd:14`,
  `client/scripts/combat_input_buffer.gd:35`), `CombatStickyTarget` owns explicit attack-move target
  memory (`client/scripts/combat_sticky_target.gd:11`, `client/scripts/combat_sticky_target.gd:22`),
  and `CommandRetargetGrace` owns latest-floor retarget replacement/expiry/dispatch state
  (`client/scripts/command_retarget_grace.gd:25`, `client/scripts/command_retarget_grace.gd:57`).
- **[Strength] Presentation offsets avoid gameplay anchors.** Movement smoothing modifies the local
  visual child and resets on large deltas (`client/scripts/movement_visual_smoothing.gd:21`,
  `client/scripts/movement_visual_smoothing.gd:30`, `client/scripts/movement_visual_smoothing.gd:38`).
  Melee lunge offsets `ModelRoot` and recovers with a tween (`client/scripts/melee_lunge_presentation.gd:21`,
  `client/scripts/melee_lunge_presentation.gd:26`), while debug state is exposed through the
  animation controller (`client/scripts/animation_controller.gd:68`, `client/scripts/animation_controller.gd:74`).
- **[Strength] Existing server intents remain the only gameplay channel.** Legal local attacks call
  `present_local_start`, then `_send_action_intent`, then local cooldown/recovery UI
  (`client/scripts/main.gd:2766`, `client/scripts/main.gd:2767`, `client/scripts/main.gd:2768`,
  `client/scripts/main.gd:2769`). Attack move still sends `move_to_intent`
  (`client/scripts/main.gd:2791`).

## 2. Technical

- **[Strength] Broad client proof is green.** Post-loop `SCENARIO=all` reported 85 passed / 0 failed.
  The dedicated movement-feel scenarios cover input buffering, sticky targeting, hit stop, movement
  smoothing, command retarget grace, and melee lunge (`tools/bot/scenarios/client/77_input_buffering.json:16`,
  `tools/bot/scenarios/client/82_melee_lunge_micro_step.json:12`).
- **[Strength] The mercenary recovery flake now has a concrete fix and unit proof.** Lost events
  remove stale companion entities before syncing the bar (`client/scripts/mercenary_panel_bridge.gd:42`,
  `client/scripts/mercenary_panel_bridge.gd:44`), and `test_mercenary_panel.gd` asserts that the
  bridge removes the stale companion (`client/tests/test_mercenary_panel.gd:79`,
  `client/tests/test_mercenary_panel.gd:82`).
- **[Med] Client feel constants are still scattered through code.** The specs intentionally documented
  code-owned microconstants for v299-v301, but the implementation now has timing/distance constants
  in `CombatInputBuffer`, `CommandRetargetGrace`, `MovementVisualSmoothing`, and
  `MeleeLungePresentation` (`client/scripts/combat_input_buffer.gd:4`,
  `client/scripts/command_retarget_grace.gd:4`, `client/scripts/movement_visual_smoothing.gd:4`,
  `client/scripts/melee_lunge_presentation.gd:5`). The project policy says presentation tuning
  belongs in shared data unless code ownership is explicitly justified (`AGENTS.md:82`).

## 3. Maintainability

- **[Strength] New domains mostly avoided new large files.** The new helper scripts are small and
  importable; `main.gd` stayed at 5,810 lines, below its grandfathered allowance.
- **[Med] `main.gd` remains the coordination bottleneck.** It still owns helper instances, input
  tick order, floor-click dispatch, attack buffering, sticky targeting, and bot debug state
  (`client/scripts/main.gd:241`, `client/scripts/main.gd:246`, `client/scripts/main.gd:2413`,
  `client/scripts/main.gd:2415`, `client/scripts/main.gd:5308`, `client/scripts/main.gd:5309`).
  The next touched movement/combat input path should move orchestration behind a narrower client
  input/presentation service.
- **[Low] Bot scenario runner remains large.** `client/scripts/bot_scenario_runner.gd` is now 1,653
  lines. v295-v301 added bot steps without splitting the runner, so future bot framework work should
  continue moving step logic into focused handler files.

## 4. Documentation

- **[Strength] Visual verification commands are present.** Each new as-built names a
  `make bot-visual scenario=...` command for the corresponding movement/combat feel slice.
- **[Strength] The final proof is documented honestly.** `PROGRESS.md` records the raw `make ci`
  Docker ownership limitation and the equivalent server-backed continuation (`PROGRESS.md:28`).

## Top 5 client refactors

1. Add a small data-backed presentation-feel config, or a single documented code-owner module, for
   input buffer duration, retarget grace, movement smoothing, hit-stop timing, and lunge distance /
   recovery before more tuning values spread across helpers.
2. Extract the next touched movement/combat input orchestration out of `main.gd` into a focused
   input/presentation coordinator.
3. Keep adding client bot debug state for presentation features, but keep the state assembly out of
   `main.gd` when a helper can own it.
4. Split the next touched `bot_scenario_runner.gd` step family into a handler/test pair.
5. Keep production VFX/audio/art polish as feature slices, not opportunistic refactors.
