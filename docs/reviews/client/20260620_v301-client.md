## Summary

The client remains a thin renderer and input/debug surface. v301 improved dungeon floor/wall visual
depth through the existing procedural material factory while preserving `StandardMaterial3D`, the
wall layout protocol, and all gameplay behavior. The prior v284 client-bot flake risk is materially
better: the full client bot batch now reports 81 passed / 0 failed.

The biggest client risk is still scale. `main.gd`, the bot scenario runner, and large UI panels are
far above target size, so future client work should keep using focused factories/controllers and
avoid adding more debug-state keys directly into `main.gd`.

## 1. Architecture

- **[Strength] v301 presentation stayed client-only and code-native.** Dungeon ground keeps a
  generated albedo texture and gains normal mapping only on dungeon levels
  (`client/scripts/ground_wall_factory.gd:34`, `client/scripts/ground_wall_factory.gd:43`). Town
  ground stays on the simple material path.
- **[Strength] Cave wall material creation is centralized without changing wall metadata.**
  `WallRenderer` delegates standard wall material construction to the factory when available
  (`client/scripts/wall_renderer.gd:115`), while wall nodes still preserve id/source/kind metadata
  (`client/scripts/wall_renderer.gd:103`).
- **[Strength] The model viewer catalog duplication noted in v284 is mostly resolved.** Both Python
  and Godot now consume the generated `shared/assets/model_preview_catalog.v0.json`
  (`tools/assets/model_catalog.py:35`, `client/scripts/model_viewer.gd:77`), and the Python test
  asserts the repo catalog matches source data (`tools/assets/test_model_catalog.py:118`).

## 2. Technical

- **[Strength] The full client bot sweep is green.** The v301 CI run passed all 81 client scenarios,
  including `elite_objective_minimap_pin`, `mercenary_recovery_ui`, `line_of_sight_blocker_shadow`,
  and `wall_floor_shader_polish`.
- **[Strength] Bot assertion precision is less brittle.** Discovery minimap float bounds now use a
  small epsilon for min/max checks (`client/scripts/bot_assertion_handlers.gd:10`,
  `client/scripts/bot_assertion_handlers.gd:391`), which fixes exact-bound float noise without
  weakening required visibility/marker assertions.
- **[Med] Scenario timing remains an active client-bot maintenance area.** The final CI pass needed
  a wider post-loss mercenary panel wait (`tools/bot/scenarios/client/49_mercenary_recovery_ui.json:27`)
  and a larger elite-objective route budget (`tools/bot/scenarios/68_dungeon_elite_side_objective.json:26`).
  These are now green, but the scenario catalog should keep favoring semantic eventual assertions
  over tight elapsed budgets.

## 3. Maintainability

- **[Strength] v301 followed the focused-factory pattern.** `ground_wall_factory.gd` is 279 lines
  and `wall_renderer.gd` is 295 lines after the material polish; the already-large
  `test_item_visuals.gd` was not touched.
- **[Med] `main.gd` remains the dominant client coordination risk.** It is still 5,813 lines, and
  `get_bot_state()` continues to aggregate many UI/debug surfaces. The next feature that needs new
  bot-visible state should prefer a focused `*_debug_state` helper or bridge.
- **[Low] Bot assertion code is still broad but under the ratchet.** `bot_assertion_handlers.gd`
  remains under 600 lines at 472, but minimap, audio, panel, fog, wall, and gameplay assertion
  domains share the same file. Future assertion additions should keep extracting helper files.

## 4. Documentation

- **[Strength] v301 records precise visual proof.** The as-built names the direct client bot command
  for `wall_floor_shader_polish` and the final full-CI proof.
- **[Low] Client bot scenario budgets are documented only in individual JSON files.** A short
  scenario authoring note for generated-dungeon route budgets would reduce future repeat fixes.

## Top 5 client refactors

1. Split the next touched `main.gd` bot/debug-state aggregation into a focused helper.
2. Extract the next touched assertion domain from `client/scripts/bot_assertion_handlers.gd`.
3. Add or update bot-scenario authoring guidance for long generated-dungeon route budgets and float
   tolerances.
4. Keep floor/wall/water/hole/obstacle material work in `GroundWallFactory`/`WallRenderer`, not
   `main.gd`.
5. Keep production dungeon art/VFX as future feature slices, not opportunistic refactors.
