## Summary

The thin-client boundary held rigorously: every new feature (water, holes, obstacle variety,
line-of-sight blocker shadows, leap arc, shader polish) is strictly presentation. Occlusion truth,
hazard pathability, and leap landing all come from server-provided wall payloads and events; the
client only decides geometry and materials. The v301 combat-feel centralization (`8a817050`) is a
genuine win — a real single owner (`combat_feel_config.gd`) with all consumers re-pointed at it. The
serious finding is a test-gating gap: the two headline unit tests for this batch plus the v301
combat-feel unit tests exist on disk but are **never registered in `scripts/client_smoke.sh`**, so they
do not run in `make client-unit` or CI — only the glob-discovered bot scenarios actually gate these
features. `main.gd` remains a 5,810-line bottleneck that the batch grew further by adding leap-visual
orchestration inline, and six grandfathered files drifted up with no baseline lowered.

## 1. Architecture

- **[Strength] LoS occlusion truth is server-owned.** `_wall_blocks_line_of_sight()` honors an explicit
  `blocks_line_of_sight` flag from the wall payload, falling back to `kind == "wall"`
  (`client/scripts/fog_of_war_overlay.gd:479`). `wall_renderer.gd:84` only propagates the flag when the
  source payload carries it; water/hole/rubble carry `kind != "wall"` and no flag, so they correctly
  default to non-occluding. The shadow-polygon math is pure screen-space silhouette projection.
- **[Strength] Occluder feed is server-state-driven.** `sync_fog_overlay()` feeds the overlay from
  `current_wall_layout` and a closed-door occluder layout filtered on server-authoritative
  `rec.state == "closed"` (`client/scripts/interactable_rules_loader.gd:46`,
  `client/scripts/interactable_rules_loader.gd:57`). No client-side collision or pathability.
- **[Strength] Leap is presentation-only.** `_play_leap_visual()` tweens `player_anchor` along an arc to
  the server event position; `_finish_leap_visual` snaps back to `predicted_pos`
  (`client/scripts/main.gd:1987`, `client/scripts/main.gd:2006`). The server owns the teleport outcome;
  obstacle-crossing legality is not evaluated client-side.
- **[Strength] Combat-feel centralization is a clean owner, not scatter.** `combat_feel_config.gd` is a
  real `class_name CombatFeelConfig extends RefCounted` constants module; all five consumers reference
  `CombatFeelConfigScript.<CONST>` (`client/scripts/combat_input_buffer.gd:5`,
  `client/scripts/command_retarget_grace.gd:5`, `client/scripts/melee_lunge_presentation.gd:6`,
  `client/scripts/movement_visual_smoothing.gd:5`). No literal duplication remains.

## 2. Technical correctness

- **[Strength] Delta-access discipline clean (rule #8).** No bracket access on delta dicts in the
  touched files; every payload read uses `.get(key, default)` — e.g.
  `client/scripts/fog_of_war_overlay.gd:481`, `client/scripts/wall_renderer.gd:34`.
- **[Strength] Shared-data singleton pattern followed (rule #7).** `combat_feel_config.gd` and
  `fog_of_war_overlay.gd` use `class_name`; `ground_wall_factory.gd:261` uses the lazy
  `_ensure_*()` load guard. No Godot-autoload dependency in touched scripts.
- **[Strength] Shader/material setup correct.** The fog shader is self-contained `canvas_item` code with
  null-guarded param pushes; `wall_renderer.gd` material paths defensively check
  `_ground_factory != null and has_method(...)` before calling.
- **[Low] One latent fragility:** `_update_shader` guards `get_viewport() != null`
  (`client/scripts/fog_of_war_overlay.gd:235`) but `_project_target` / `_projected_radius` fallbacks
  assume non-null without the same guard. Has not crashed (a viewport always exists under `SceneTree`),
  but the asymmetry is a foot-gun.

## 3. Test quality

This is the weakest axis, on two counts.

- **[High] The headline unit tests are never run in CI.** `scripts/client_smoke.sh` registers each unit
  test explicitly via `run_gate`, not by glob. `test_factories.gd` and `test_fog_of_war_overlay.gd`
  (this batch) and the v301 `test_movement_visual_smoothing.gd`, `test_melee_lunge_presentation.gd`,
  `test_command_retarget_grace.gd` are **not registered** — all five files exist under `client/tests/`
  but execute zero times in `make client-unit` / CI. The as-built docs only document a manual
  `godot --headless --script …` invocation (`docs/as-built/v307_line-of-sight-blockers.md:22`,
  `docs/as-built/v308_wall-floor-shader-polish.md:24`), and the plan templates left placeholder
  `tests/n.gd` filenames — evidence the wiring step was skipped. The features are still covered by the
  glob-discovered bot scenarios (77/78), so this is orphaned regression protection, not untested
  features — but it is dead weight that rots silently. **(Verified: `rg` over `client_smoke.sh` finds
  none of the five; all five files present on disk.)**
- **[Med] Scene-tree ops in `extends SceneTree` tests (rule #9 tension).** Both new tests do
  `get_root().add_child(...)` + `await process_frame` (`client/tests/test_fog_of_war_overlay.gd:34`,
  `client/tests/test_factories.gd:68`). Rule #9 forbids this for headless `extends SceneTree` tests.
  In fairness ~19 of 52 existing tests already do it, so the batch follows an established
  (rule-violating) house pattern. The rule should be either enforced or amended to sanction
  "node-render integration tests."
- **[Strength] Where coverage lives, it is good.** The `_fog_of_war_mismatch` assertion handler supports
  `_min`/`_max`/exact-with-tolerance (`client/scripts/bot_assertion_handlers.gd:356`), aligned with the
  semantic-range test policy; the unit tests' assertions (if they ran) are mostly substantive
  (water/hole/rubble skip shadows, out-of-range culling, wall-top silhouette geometry).

## 4. Maintainability

- **[Med] `main.gd` is still the bottleneck and grew.** 5,810 lines, up from baseline 5,788 (+22,
  3 lines from the gate; `.maintainability/file-size-baseline.tsv:7`). Factory/renderer logic was
  correctly extracted to focused files, but the *input/presentation orchestration* the v301 review
  flagged (leap tween at `client/scripts/main.gd:1987`, fog sync calls) was added inline — the
  coordinator got bigger, not smaller.
- **[Med] Six grandfathered files grew, zero baselines lowered.** `main.gd` +22,
  `bot_scenario_runner.gd` 1,629 → 1,653 (+24, one edit from a CI failure), `inventory_panel.gd` +22,
  `shop_panel.gd` +6, `market_panel.gd` +7, `skills_panel.gd` +2. The baseline TSV was last touched at
  v293. Each passes the ratchet only by consuming the incidental-drift budget — exactly what CLAUDE.md
  rule #3/#4 warns against.
- **[Strength] New domains correctly started in their own files.** `combat_feel_config.gd` (~15),
  `fog_of_war_overlay.gd` (~519), `ground_wall_factory.gd` (~279), `wall_renderer.gd` (~295) are all
  well under 600 and single-responsibility — the new logic did not get dumped into `main.gd`.

## Prior v301 client recs — confirmation

- **v301 rec #1/#2 (central owner for feel constants) — RESOLVED, with a caveat.** `8a817050` created a
  genuine single owner (`client/scripts/combat_feel_config.gd`) and re-pointed all consumers; no literal
  duplication remains. **Caveat:** it is code-owned (`const`s), not data-backed in
  `shared/rules/main_config.v0.json`. Per ADR-0007 these are client-only presentation-feel values, so
  code ownership is defensible — but the commit did not leave the explicit "why code ownership is right"
  note the Data-Driven Configuration Policy asks for.
- **v301 rec (extract movement/combat input orchestration out of `main.gd`) — OPEN / slightly worse.**
  The batch touched `main.gd` (leap visual, fog sync, wall-render hooks) and grew it +22; the
  orchestration the rec called out was not extracted.

## Top 5 client refactors

1. **Wire the five orphan unit tests into `scripts/client_smoke.sh`** (`run_gate` lines for
   `test_factories.gd`, `test_fog_of_war_overlay.gd`, `test_movement_visual_smoothing.gd`,
   `test_melee_lunge_presentation.gd`, `test_command_retarget_grace.gd`). Highest-value, lowest-risk
   fix — they currently provide zero regression protection.
2. **Extract the leap/charge visual orchestration (`main.gd:1987`–~2050) into `skill_visual_presentation.gd`** —
   self-contained tween logic with a narrow `player_anchor` + `predicted_pos` dependency; pays down the
   v301 main.gd rec and lowers the baseline.
3. **Resolve the rule #9 tension explicitly** — either refactor the two tests to avoid
   `get_root().add_child`, or amend CLAUDE.md rule #9 to sanction node-render integration tests (since
   ~19/52 tests already do this). Stop pretending a routinely-ignored rule is enforced.
4. **Update `docs/CODEMAP.md`** with fog/LoS, obstacle-rendering, and combat-feel-config rows so the
   documented entry point stops misdirecting to `main.gd`.
5. **Add the data-driven-config decision note for `combat_feel_config.gd`** (one-paragraph header or
   PROGRESS.md note: client-only presentation-feel constants owned in code per ADR-0007). Closes the
   policy gap cheaply.
