## 1. Architecture

**[Strength]** `HeroLightSource` (`client/scripts/hero_light_source.gd`, 105 lines) is entirely static, has no scene-tree coupling or `@onready` vars, and touches nothing outside rendering geometry. Textbook ADR-0001 D2 compliance: the server is not consulted for mesh bounds or light height.

**[Strength]** `DiscoveryMinimapState` (`client/scripts/discovery_minimap_state.gd`, 253 lines) is a standalone `RefCounted` that holds all per-level fog-of-war bookkeeping that previously lived inline in `DiscoveryMinimap`. The extraction is architecturally complete — it has its own `class_name`, is independently preloaded, and `DiscoveryMinimap` delegates to it via `_state_tracker`. `test_discovery_minimap.gd` imports it directly without touching `DiscoveryMinimap`.

**[Strength]** `FogOfWarOverlay` owns the `OmniLight3D` lifecycle (`_setup_point_light`, `_reparent_point_light`, `_sync_point_light`). The light position is computed by delegating to `HeroLightSource.local_light_position()` / `estimate_world_height()`. Correct direction of dependency: overlay knows about the helper, not the reverse.

**[Med]** `_sync_discovery_minimap()` at `main.gd:5759` passes `_last_facing_direction` — a client-side input accumulation variable — directly into the minimap. This is presentation-local state being routed through the god-coordinator. The call is a one-liner and not structurally wrong, but it adds one more tentacle from `main.gd` into a downstream component that could receive it through a richer sync context object in a future cleanup.

**[Low]** `discovery_minimap.gd:226` calls `_draw_arrow_marker()` but there is no headless test that verifies the arrow geometry (tip, base, direction) — only the player marker center and the `player_facing_x/y` fields in state are asserted. The draw logic is only indirectly exercised by the smoke gate.

## 2. Technical

**[Strength]** `HeroLightSource.estimate_world_height()` handles three fall-through cases cleanly: mesh present and in tree → global transform; mesh present but not in tree → local scale approximation; no mesh → anchor + min_height. All paths have explicit guard clauses.

**[Strength]** `_reparent_point_light()` at `fog_of_war_overlay.gd:582` was added when `bind()` is called. Previously a re-bind (e.g. camera mode switch) left the `OmniLight3D` parented to the old target. The fix correctly removes from the old parent before re-parenting and guards on `_point_light == null`.

**[Strength]** Wall seam fix in `wall_renderer.gd:135`: `WALL_FLOOR_SEAM_OVERLAP = 0.08` is a named constant, not a magic number. The overlap is conditional on `_dungeon_presentation_active()` — town walls are unaffected. The test at `test_factories.gd:83` asserts `wall.size.y == dungeon_ceiling_height() + 0.08`, which pins this constant without hardcoding the base height.

**[Med]** `fog_presentation_loader.gd` now has both `height_offset` (legacy) and `height_fraction` + `min_height` (new) in `_default_config()`. `HeroLightSource.estimate_world_height()` at line 4 reads `min_height` with fallback to `height_offset`, so old configs degrade. However `_merge_defaults()` in `fog_presentation_loader.gd` preserves both keys, meaning any production `fog_presentation.v0.json` that omits `height_fraction` silently uses the default 0.55 without any log or validation warning. This is a latent misconfiguration risk, not a crash.

**[Low]** `HeroLightSource._mesh_instances()` at line 92 is a recursive tree walk. For a typical character rig this is fine (shallow), but there is no depth limit. Pathological scene trees (nested LOD groups, editor leftovers in a GLB) would result in unbounded recursion. Not a real risk in the current asset pipeline, but worth a comment.

**[Low]** `_hero_light_world_height()` is defined in `fog_of_war_overlay.gd:419` and also called by `get_debug_state()` at line 288. The debug state payload now exposes `hero_light_height`, which is good for diagnosing camera projection bugs in perspective mode. No concern — just noting it is intentional.

## 4. Documentation

**[Low]** `hero_light_source.gd` has a single file-level comment (`## Estimates hero-attached light height from the visible character mesh bounds.`) but the two main public entry points (`estimate_world_height` and `local_light_position`) have no doc comments explaining the difference between them. The distinction (world Y coordinate vs. local Vector3 relative to the visual node) is not obvious from the name alone.

**[Low]** `discovery_minimap_state.gd` has no file-level or function-level comments. The `update()` method has six parameters including the optional `player_facing`; callers who pass `Vector2.ZERO` get a silently defaulted north-facing — this default behavior is not documented at the call site or in the function signature.

**[Strength]** `fog_of_war_overlay.gd:133` has a block comment explaining why `OmniLight3D` is needed in perspective mode (canvas fog is transparent). This is the right level of documentation for a non-obvious architectural choice.

## Top 5 Client Refactors

**1. [High] Extract `main.gd` movement-intent orchestration into a coordinator.**
`main.gd` is at 6,084 lines / 447 functions. The next extraction target should be the movement + attack input cluster (functions around lines 2900–4200 that reference `_last_facing_direction`, `CombatReachScript`, `DirectionalAttackInputScript`, `_channel_skill_input`). These functions share a narrow set of state variables and can be isolated into a typed input context without touching the server protocol.

**2. [Med] Add `fog_of_war_overlay.gd` to the maintainability baseline or extract the `OmniLight3D` management block.**
The file crossed 600 lines in v331. The `_setup_point_light`, `_reparent_point_light`, `_sync_point_light` cluster (lines 582–640) is a natural extraction candidate as a `HeroPointLight` helper class, which would also simplify `fog_of_war_overlay.gd` and make the light lifecycle unit-testable independently.

**3. [Med] Document `HeroLightSource` public API with `##` doc comments.**
`estimate_world_height()` returns a world-space Y coordinate; `local_light_position()` returns a local-space Vector3 relative to the character visual. Without doc comments, callers (currently `fog_of_war_overlay.gd`) can easily misuse these when the light is reparented. Takes 10 minutes, pays down future confusion.

**4. [Low] Add a `fog_presentation.v0.json` schema validation for `height_fraction` and `min_height`.**
`shared/assets/fog_presentation.v0.schema.json` should enumerate `height_fraction` and `min_height` as required keys now that `height_offset` is legacy. `make validate-shared` will then catch misconfigured production JSON before it reaches the client.

**5. [Low] Add a headless draw-geometry test for `_draw_arrow_marker()` in `test_discovery_minimap.gd`.**
The player marker direction arrow introduced in v331 (`discovery_minimap.gd:226`) is pure draw logic — it is not covered by any assertion beyond the `player_facing_x/y` state fields. A test that calls `_draw_player_marker()` on a controlled node and inspects the drawn polygon vertices (or at minimum confirms no crash for edge-case facings like `Vector2.ZERO`, `Vector2.UP`, `Vector2(-1, 0)`) would close the gap.
